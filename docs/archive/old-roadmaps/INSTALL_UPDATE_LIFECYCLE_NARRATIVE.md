## 1. Install/update operator entrypoint (`install-update.ps1`) — as-built detail

An **install/update operator entrypoint** `scripts/install-update.ps1` exists, with read-only `inspect` / `verify` modes (Batch 1, committed `8ddb04c`) and an **approval-gated `update-source` mutation mode** (Batch 2). All modes emit a fixed-vocabulary final status plus a stdout JSON report (a strict field-name subset of the `run.json` shape). `inspect` / `verify` write no file (stdout/stderr only). `update-source` rewrites the InstallArea payload (`current/` + `install.json` + `payload-manifest.json` + `payload-marker.json`) via the canonical pipeline (`New-InstallPipelineTuple` action=`update-source` → `Invoke-InstallPipelineDispatch`), preserving `installedHead`/`installedAt` and updating `lastUpdatedHead`/`lastUpdatedAt`, then runs post-apply canonical `Invoke-InstallPipelineVerify` + activation byte-identity + run-scoped work-area cleanup + optional operational smoke. The payload mutation uses **command-implied approval** (Batch 2.1): the operator's explicit `update-source` invocation on an existing identity-consistent install — under an explicit user update instruction + a read-only plan-report — is the approval surface, so a normal noninteractive Claude Code shell is not blocked by a mandatory terminal prompt. The hard guards (source-cut / missing-or-invalid metadata / identity ambiguity / destination-not-an-existing-install / unresolved HEAD) still stop the mutation (`failed`); command-implied approval applies only to update-source of an existing install (never fresh install, new destination, activation apply, or a source-cut override). An optional `-ConfirmInteractive` switch enables the exact two-choice (Yes/No) terminal selector for direct-terminal use (default OFF; if requested without a terminal it aborts rather than auto-applying). Activation surfaces (managed blocks / skill mirror) are **verified by byte-identity, not written**, by this entrypoint — a byte-identical surface is a no-op, a drifted one yields `activation_pending`; actual activation apply remains the separate `scripts/activate-global.ps1` + `scripts/apply-managed-block.ps1` step. Activation drift is detected by byte-identity (Claude/Codex managed blocks via marker-bounded content equality, skill mirror via SHA-256), not mere path existence; the Codex managed-block surface honors `AGENTS.override.md` precedence. The final-status vocabulary, `run.json` evidence contract, operational-smoke contract, and two-choice approval-selector contract are codified in `INSTALL.md` §13; the canonical apply entrypoint is codified in `INSTALL.md` §7.1.1; the deterministic-narrow-entrypoint allowance (vs the still-out-of-scope productization/framework class) is codified in `INSTALL.md` §11 (b). The canonical verifier `Invoke-InstallPipelineVerify` enforces the complete `INSTALL.md` §4/§5 mode-conditional metadata schema for both modes (git-url: `repoUrl` non-empty + `sourcePath`/`toolRoot` empty; local-clone: `sourcePath`/`toolRoot` non-empty + `repoUrl` empty). Still future scoped work: actual `run.json` write to the global install log path, and snippet routing to this entrypoint.

## 2. Post-MVP install lifecycle maintenance phases (1–4a)

Sequenced by dependency/risk, not a vague backlog.

- **Phase 1 — observability / status correctness** (commit `9d02810`): `update-source` stdout now emits real post-apply diagnostics instead of misleading `false`/`null` defaults, empty `leftoverPaths` serializes as `[]`, the `-Json` stdout/stderr contract and the `activation_pending`/activation-only-`verify_failed` "payload-OK-with-followup" semantics are documented (`INSTALL.md` §13.2/§13.4/§13.7/§13.11).
- **Phase 2 — bootstrap / name-based update discoverability** (docs-only): `INSTALL.md` §7.1 carries a self-contained name-based update quickstart + a legacy-bootstrap note (an installed entrypoint may predate `update-source`; clone the latest source and run the cloned `scripts/install-update.ps1`, not the installed copy) + an operator bootstrap-clone cleanup rule (distinct from the script-internal source-cache cleanup), and a new `INSTALL.md` §7.2 flow matrix distinguishes fresh/full install vs name-based update vs activation refresh by payload/activation mutation, approval model, entrypoint, and terminal status. No behavior change. A deployable pointer file (`UPDATE.txt`/`UPDATE.md`) was evaluated and deferred (docs-only chosen).
- **Phase 3 — source identity / acquisition ergonomics** (docs-first, no code change): `INSTALL.md` §7.1 documents that name-based update should omit `-RepoUrl` and acquire per `install.json.repoUrl` (the source-cut guard compares identity fields without normalization, so an equivalent-but-differently-spelled URL — `.git`/trailing-slash; mere casing is compared case-insensitively and does not trip — can trip a false-positive `failed`; a real source-cut override stays outside command-implied approval), that the bootstrap clone and the update-source internal source-cache acquisition are a deliberate double acquisition with distinct cleanup ownership (no silent cache reuse; `-AcquisitionClonePath`/cache-reuse deferred), and that an installed copy already having `update-source` does not change the cloned-latest-script rule. The source-cut hard guard itself is unchanged.
- **Phase 3.5 — managed installed-root README** (pipeline code + docs): the canonical pipeline now materializes a managed operator landing page at `<InstallArea>/README.md` (sibling of `current/`) from an in-payload template `templates/install-root/AI_HARNESS_TOOLSET_ROOT_README.md`, on both fresh install and update-source; `Invoke-InstallPipelineVerify` verifies it by SHA-256 byte-identity against the in-payload (manifest-covered) template, so missing/stale/corrupted README surfaces as `verify_failed` (template-conditional: skipped when the payload lacks the template, e.g. legacy source). The installed README is a landing page only; the operative update contract remains the latest source clone's `INSTALL.md`. Supersedes the Phase 2 deferred deployable-pointer item (chosen as a managed `README.md` rather than `UPDATE.txt`).
- **Phase 3.5.1 — root README reinstall-first verification policy** (pipeline + entrypoint + docs): the managed root README is treated strictly as a **canonical output** of a normal install / payload-rewriting update-source, not a self-healing target. The Phase 3.5 in-place self-heal (`Repair-RootReadmeInPlace`) is **removed** — it conflicted with the §9/§19 reinstall-first model. A missing/stale/corrupt root README on an otherwise-current install is now an **install-integrity failure**: `verify` reports `verify_failed` and `inspect` classifies it as `inspect_payload_drift`, with both modes' `reasons` pointing to reinstall-first recovery; the integrity check is single-sourced via `Get-InstallPipelineRootReadmeState`. No status/exit-code vocabulary change.
- **Phase 3.6 — activation follow-up UX** (human-output + activation tooling + docs; no status/exit-code vocabulary or apply-behavior change; driven by mainpc/notebookpc global dogfood `6dc5e2f`→`bceb7a2`): (1) `update-source` `activation_pending` human output reads as a follow-up, not a hard failure (`followUpRequired=activation` / `payload=ok` / `result=INCOMPLETE`; machine status stays `activation_pending`, exitCode stays 1, JSON unchanged); (2) prints the exact copy-paste next command using `<InstallArea>\current\scripts\activate-global.ps1`; (3) `scripts/apply-managed-block.ps1` dry-run prints a compact change summary by default (`-ShowFullDiff` for the legacy dump); (4) `-InstallArea` clarified as the install ROOT, and passing `current/` yields `inspect_mode_unknown` with a "did you mean its parent" hint.
- **Phase 3.7 — cleanup / backup UX micro-polish** (docs/test only; driven by mainpc/notebookpc dogfood of `bceb7a2`→`38102b8`): (1) `INSTALL.md` §7.1 "Operator bootstrap clone cleanup 규칙" reinforced — auto-deleted after a successful update/activation/verify with no "delete it?" prompt; leftover path + reason only in the three exception cases. (2) `INSTALL.md` §10 documents the `.amb-backup` rollback-sidecar lifecycle from the actual `scripts/apply-managed-block.ps1` behavior — created immediately before the write, removed on a successful apply, persisting only abnormally; a new apply fails fast on an already-present `.amb-backup`. (3) installed root README gets short operational notes. No status/exit-code/JSON change.
- **Phase 4a — activation apply orchestration is complete** (source commit `0171743414a1caffbfff6d7bbb4d1af5cfcc27dd`). Aligned the activation **apply** path with what install-update **verifies**: `scripts/activate-global.ps1` now applies all three activation surfaces (Claude managed block, Codex effective managed block, review-skill mirror) through the shared resolver `scripts/lib/activation-surface.ps1` that update-source's verify path also uses, removing destination drift; the two mutation classes (managed-block marker-bounded splice with `.amb-backup`/rollback, and the skill-mirror canonical-overwrite with post-write SHA-256 verify and no backup) are orchestrated with a preflight-all-then-apply discipline and a two-state (Yes/No) interactive confirm, default dry-run. update-source itself is unchanged (still activation byte-identity verify-only). **main PC / notebook PC global dogfood completed** end-to-end: a natural-language update reached `update-source`, surfaced `activation_pending`, ran the `-DryRun` preview across the 3 surfaces, took an explicit activation apply (`activate-global -Apply`), and ended in a final `verify_pass`; bootstrap clone cleaned up with no leftover, no `.amb-backup` remained, review-skill mirror byte-identical.
