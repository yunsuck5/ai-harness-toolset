## Accepted residual risks (LTS maintenance)

The following residual risks were surfaced by the LTS closeout disposition analysis (repo-sibling: `polishing/review/review_polishing_disposition_analysis_20260527.md`) and are explicitly accepted as part of LTS maintenance posture. Reopen requires concrete new evidence beyond the current pattern.

- **H1 preamble / Counter-argument surface alignment** (closed by Stage 4-R1) — the prior dual-surface tension (H1 preamble strongest-case-against pointing to `## Notes` vs `REVIEW_RESULT_CONTRACT.md §3c` making `## Counter-argument` the dedicated surface) was disclosed in Phase 2 Implementation Plan §11.2.1 and user-accepted at Batch 1A entry. Stage 4-R1 (this batch) aligned the H1 reviewer-mode preamble in `scripts/review-run.ps1` to direct strongest-case-against substance to `## Counter-argument`; `## Counter-argument` remains optional / strongly-recommended / non-parser (no Option A parser-required escalation). Reopen criteria for Option A remain in `REVIEW_RESULT_CONTRACT.md §3c.5`.
- **Operator narrow-scope over-confidence pattern** (recurrence; dual nature — operator failure mode + review-system effectiveness evidence) — Batch 1A pass-02 within-file enumeration, IDEAS governance citation provenance, and v2/v3 polishing-inventory count drift are documented recurrences. In each case Codex independent review mechanically caught the operator scope-narrowing before it could land as final accepted state (via `Get-ChildItem` for filesystem count, `rg -n` for citation provenance, and grep for within-file enumeration); the recurrence is operator-side failure mode while the catch is review-system effectiveness evidence — both framings hold simultaneously in LTS maintenance. Residual: operator can still create wrong inputs before Codex review runs. Mitigation: Codex independent review (primary) + operator-side discipline codified in `snippets/claude-skills/ai-harness-review/SKILL.md` step 4 (citation provenance verify before quoting + mechanical count check + grep-based exhaustiveness) + `REVIEW_RESULT_CONTRACT.md §5a.4` retraction protocol when discovered ex-post. No script-side enforcement (governance plan D12 not-implementable-yet).
- **Bash chain final-exit-code false-fail** (operator reporting convention) — when a `;`-chained shell line ends with a command whose non-zero exit is informational (e.g., absence-check `ls` of a directory expected to be missing), the chain's final exit code can mask the success of the preceding `review-verify` call. Harness-level mitigation is out of toolset scope (the toolset does not own the shell-chain reporter). Operator convention: when reporting a `review-verify` result, do not chain it before an informational command whose non-zero exit would override the success exit.
