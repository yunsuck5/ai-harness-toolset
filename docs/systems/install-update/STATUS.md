## Current state

- Current default adoption shape: **shared / global stable runtime ToolRoot (channel 3)** — lifecycle scripts run from `%USERPROFILE%\.claude\ai-harness-toolset\current`, resolved per invocation; runtime output under the target's `<project-root>/log/`. Legacy project-local copy mode (channel 5) is supported for backward compatibility only.
- Install/update automation exists as a temp-only local-clone + git-url(local-git) skeleton with a payload integrity manifest + completeness marker; recovery policy is **reinstall-first** (detection-only, deterministic overwrite from trusted source identity = resolved commit SHA). Deferred remainder: `DEFERRED.md`.
- An **install/update operator entrypoint** `scripts/install-update.ps1` exists, with read-only `inspect` / `verify` modes (Batch 1, committed `8ddb04c`) and an **approval-gated `update-source` mutation mode** (Batch 2). All modes emit a fixed-vocabulary final status plus a stdout JSON report (a strict field-name subset of the `run.json` shape). `inspect` / `verify` write no file (stdout/stderr only). `update-source` rewrites the InstallArea payload (`current/` + `install.json` + `payload-manifest.json` + `payload-marker.json`) via the canonical pipeline (`New-InstallPipelineTuple` action=`update-source` → `Invoke-InstallPipelineDispatch`), preserving `installedHead`/`installedAt` and updating `lastUpdatedHead`/`lastUpdatedAt`, then runs post-apply canonical `Invoke-InstallPipelineVerify` + activation byte-identity + run-scoped work-area cleanup + optional operational smoke. The payload mutation uses **command-implied approval** (Batch 2.1): the operator's explicit `update-source` invocation on an existing identity-consistent install — under an explicit user update instruction + a read-only plan-report — is the approval surface, so a normal noninteractive Claude Code shell is not blocked by a mandatory terminal prompt. The hard guards (source-cut / missing-or-invalid metadata / identity ambiguity / destination-not-an-existing-install / unresolved HEAD) still stop the mutation (`failed`); command-implied approval applies only to update-source of an existing install (never fresh install, new destination, activation apply, or a source-cut override). An optional `-ConfirmInteractive` switch enables the exact two-choice (Yes/No) terminal selector for direct-terminal use (default OFF; if requested without a terminal it aborts rather than auto-applying). Activation surfaces (managed blocks / skill mirror) are **verified by byte-identity, not written**, by this entrypoint — a byte-identical surface is a no-op, a drifted one yields `activation_pending`; actual activation apply remains the separate `scripts/activate-global.ps1` + `scripts/apply-managed-block.ps1` step. Activation drift is detected by byte-identity (Claude/Codex managed blocks via marker-bounded content equality, skill mirror via SHA-256), not mere path existence; the Codex managed-block surface honors `AGENTS.override.md` precedence. The final-status vocabulary, `run.json` evidence contract, operational-smoke contract, and two-choice approval-selector contract are codified in `INSTALL.md` §13; the canonical apply entrypoint is codified in `INSTALL.md` §7.1.1; the deterministic-narrow-entrypoint allowance (vs the still-out-of-scope productization/framework class) is codified in `INSTALL.md` §11 (b). The canonical verifier `Invoke-InstallPipelineVerify` enforces the complete `INSTALL.md` §4/§5 mode-conditional metadata schema for both modes (git-url: `repoUrl` non-empty + `sourcePath`/`toolRoot` empty; local-clone: `sourcePath`/`toolRoot` non-empty + `repoUrl` empty). Still future scoped work: actual `run.json` write to the global install log path, and snippet routing to this entrypoint. (Folding activation apply into this entrypoint's update flow was the Phase 4b one-shot candidate, **now RETIRED — see `BACKLOG.md` IU-B-07 [RETIRED]; removed from the post-MVP work list, reopen only by explicit decision**. The standalone activation apply orchestration itself shipped in Phase 4a as `scripts/activate-global.ps1`, but this entrypoint still does not apply activation and a natural-language update still does not imply one — that separate-explicit-step design is the retained intended behavior.)
