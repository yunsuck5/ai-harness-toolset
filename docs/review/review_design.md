# review Design

## Header

**이 문서는 무엇의 Design 인가.** review 도메인 마이그레이션(batch R)의 방향 문서다 — 분산된 review 문서 표면(`docs/contracts/review/REVIEW_RESULT_CONTRACT.md` · `docs/contracts/evidence/EVIDENCE_CONTRACT.md` · `docs/policies/REVIEWER_CONFIG_POLICY.md` · `docs/policies/REVIEW_EFFORT_GUIDE.md` · `docs/systems/review/` 13파일)을 docs-working-model 의 도메인 구조(`docs/review/`)로 수렴시키는 변경의 why / what / owner-surface 결정을 담는다.

**이 체인이 끝나면 무엇이 되는가.** `docs/review/review_spec.md`(목표 상태 명세, 구현과 1:1 동기화)와 `docs/review/review_backlog.md` 가 live 가 되고, 구계열 review 문서·routing 항목은 retire 되어 review 도메인이 자기 폴더 안에서 닫힌다. glossary pending 2종(`review-support naming` · `evidence contract absorption`)이 close 된다.

**이 문서가 아닌 것.** Plan 아님 · Spec 아님 · Work Packet 아님 · review scripts/skill/templates/config/tests 의 behavior 변경 아님. 이 Design 은 mutation/commit/push 승인이 아니다(1회 진술).

## 왜 바꾸는가 / 무엇을 바꾸는가

**왜.** review 도메인의 normative 사실이 네 계층(contracts 2 · policies 2 · systems 13 · active surface 기술)에 분산되어 있고, 그 형태가 모델의 다섯 산출물 분류와 비정합하다 — REVIEW_RESULT_CONTRACT 는 스스로 "specification of record(operative authority 아님)"를 선언하는데 이는 정확히 도메인 spec 의 역할이라 자리만 다른 spec-of-record 가 두 개 생길 수 있는 상태이고, `docs/systems/review/` 에는 구현 완료된 당시 lifecycle 기록(PLAN/SPEC/RECORD 류) 10종이 누적되어 있으며, STATUS 는 posture / completed ledger / narrative 가 한 파일에 있고, BACKLOG 와 별도로 IDEAS 라는 제3의 future-work 변형이 존재한다. 미이주 도메인 중 잔재가 가장 많다. single-home 원칙 하에서 이 사실들의 단일 거처는 도메인 폴더의 spec/backlog 다.

**무엇을.** ① `docs/review/` 도메인 폴더에서 review_spec 을 생산하고(아래 처분 결정에 따라 구계열 내용을 normative 문장으로 재구성), ② BACKLOG·IDEAS·STATUS 잔여 risk 를 review_backlog.md 로 통합하고, ③ 구계열 review 문서를 retire 하고, ④ review routing 을 `docs/README.md` 로 이관하고, ⑤ glossary pending 2종을 close 한다. 구현 behavior 변경은 0 — 이 변경은 문서 표면의 수렴이다.

## Owner surface model

- **behavior owner 는 변하지 않는다**: `scripts/review-prepare.ps1` / `review-run.ps1` / `review-verify.ps1` / `review-input-verify.ps1` / `review-safety-negtest.ps1`(+`scripts/lib/path.ps1` 의 layout/containment 검증)가 기계 gate·canonical layout·run-fact emission 의, `templates/review-input.md` / `templates/review-result.md` 가 input/result shape 의, `snippets/claude-skills/ai-harness-review/SKILL.md` 가 operator workflow·stance 의, `config/reviewer.json` + `config/reviewer.schema.json` 이 model/effort/categoryPolicy 값과 per-key 의미 기술의, `tests/review-*.Tests.ps1` + `tests/path.Tests.ps1` 이 지속 검증의 owner 다. review_spec 은 이들을 명세하고 대조될 뿐 operative authority 가 아니다(root *Final hard rule*).
- **흡수 고도(결정).** spec 의 1:1 단위는 normative 문장이다 — 외부 관찰 가능 행동과 소유 경계를 명세하고, 기계 세부(정확한 H2 enumeration·placeholder regex·precedence 분기표·run-fact 라인 목록)는 active surface(templates/scripts/config schema)가 소유하는 값으로 두고 spec 은 그 invariant 와 owner 만 명명한다. contract 의 절 구조·목록 형식을 형식째 복제하지 않는다(spec 은 prose mirror 가 아니다).
- **배포·active 표면의 참조 실태와 경계(결정).** 배포 payload 중 skill(`snippets/claude-skills/ai-harness-review/SKILL.md`)·snippets·templates 는 repo `docs/**` 경로를 참조하지 않는 self-contained 표면이다(retire 대상 17파일명 전체의 case-insensitive 전수 sweep 으로 확인). 반면 scripts·config·tests 는 retire 대상 경로의 **비-behavior 참조**를 가진다: `scripts/review-run.ps1` 주석 4건(REVIEWER_CONFIG_POLICY 3 · REVIEW_POLISHING_DECISION_RECORD 1) · `scripts/review-safety-negtest.ps1` 주석 1건(REVIEW_POLISHING_BATCH_A_SPEC) · `config/reviewer.schema.json` description 내 경로 참조 1건(REVIEW_POLISHING_DECISION_RECORD; 그 외 "decision record" 의미 언급들) · `tests/review-run.Tests.ps1` 주석 1건(contract §3c) · `tests/README.md` 3건(contract 2 · STDOUT_ANCHOR_TEST_PLAN 1). 이들 주석·schema description·문서 포인터의 갱신(review_spec 재지향 또는 직접 결정 문장화)을 이 batch 의 scope 에 포함한다 — 전부 비-behavior 이며, 무행동변경은 verify-ps1 + full Pester 로 입증한다. (`tests/install-update.Tests.ps1` 의 `review_polishing` 토큰은 `log/` runtime 폴더명 regex 로 docs 참조가 아니다 — 비대상.) scripts·config 는 배포 payload 이므로 이 갱신은 repo 와 글로벌 설치본 사이에 **한시적 비-behavior wording 편차**를 만든다 — 설치본 refresh 는 별도 명시 결정(이 batch 비범위)이고, review engine 은 global stable ToolRoot 를 쓰므로(review-subsystem self-modification 의 독립 엔진 규율) 이 편차가 review 운용에 영향을 주지 않는다.

## 수정 대상 (구조물별 처분 결정)

수정하는 기존 live Spec 은 없다(이 도메인의 첫 마이그레이션). 처분 판정 기준: **(A) consumer 판정** — 외부 adopter 가 review 구현 없이 준수해야 하는 입출력 계약인가(yes → 별도 contract role 존속, no → spec 흡수); **(B) current-bearing 판정** — 당시 기록 중 unique live 의미만 흡수하고 나머지는 git history; **(C) 3분리** — posture/capability → spec, open 항목 → backlog, narrative → git history.

- **`docs/contracts/review/REVIEW_RESULT_CONTRACT.md` — 흡수 후 retire (기준 A: 별도 contract role 불채택).** 판정 근거: ① 이 toolset 의 배포는 docs-free 이고 배포 표면(skill·templates·scripts·config)은 이 contract 경로를 참조하지 않는다(참조 0건 검증) — adopter 는 이 문서를 받지도 읽지도 않는다. ② adopter 가 실제로 만나는 입출력 계약(input.md 5-H2 · result.md verdict+4-H2 · three-level layout)은 deployed templates 가 seed 하고 scripts 가 기계 gate 로 강제한다 — "구현 없이 준수" 시나리오가 존재하지 않는다. ③ contract 스스로 operative authority 가 active surface 에 있는 specification-of-record 임을 선언한다 — 그 역할은 review_spec 과 동일하므로 공존은 single-home 위반이다. 따라서 auxiliary `_contract` 파일을 신설하지 않는다(rule 의 deferred 기본 유지). 흡수 내용: artifact 정체(2-file·three-level·write-once)·dual-authorship·verdict 어휘와 next-action 의무·staleness/stale-by-omission·operator stance(§5a)·script 책임 경계(§4)·evidence 경계(§3a/§3d)·report schema 의 축 분리(§6b)·validation scope 정책(§6c)·non-goals 중 durable 한 금지 경계 → review_spec 의 normative 문장으로 재구성 이주. 절→문장 대응의 line-level 대조는 Work Packet 소관.
- **`docs/contracts/evidence/EVIDENCE_CONTRACT.md` — 두 층 분리 흡수 후 retire.** evidence 는 도메인이 아니다(glossary rejected 유지). 이 contract 가 섞어 담은 두 층의 owner 를 분리해 정한다. ① **generic 자리·runtime 성격** — `log/evidence/` 가 `<ProjectRoot>/log/` runtime 트리 하위이고 gitignored·commit/snapshot 비포함이라는 사실은 evidence 고유 의미가 아니라 `log/` footprint 경계의 일반 성격이다; owner 는 기존대로 footprint 경계(install-update 도메인)와 그것을 interface 로 참조하는 각 사용 도메인이며, review_spec 은 이를 **cross-domain interface 로 참조만** 하고 재정의하지 않는다. 타 표면(docs-working-model rule 의 closeout evidence 기록처, `docs/brief/brief_spec.md` 의 대응 근거 기록처, root README 의 운영 안내)의 `log/evidence/**` 사용은 이 경로 interface 사용이지 evidence 파일 형식 의미에 대한 의존이 아니다 — retire 연동 갱신은 pointer 수준만. ② **evidence 파일 형식 규약과 의미 경계** — `<scope>/<case>/` 경로 패턴, 5-file recipe 와 single-Markdown bundle 의 양립, R1 referencing 대상의 single-bundle 한정, 의미 경계(재실행 아님·truth oracle 아님·freshness binding 아님·source-of-truth 승격 없음) — 의 정의 소비자는 review 흐름(operator 의 evidence 작성·reviewer 의 inspect·R1 referencing)이므로 review_spec 으로 흡수한다. manual capture recipe(단계별 절차·PowerShell snippet)는 튜토리얼 서사로 git history 보존(spec 은 실행 명령 시퀀스를 담지 않는다; operator-facing evidence guidance 는 배포 표면이 이미 self-contained 로 운반). 구현 단계 sweep 에서 형식 규약(②)의 비-review 소비자가 발견되면 이 분리 판정을 재확인한다(Work Packet 분류 대상).
- **`docs/policies/REVIEWER_CONFIG_POLICY.md` — 흡수 후 retire.** invariant(model/effort 는 config-driven · model 하드코드/내장 fallback 금지·결손 시 fail-fast · categoryPolicy 는 operator 명시 선택+기계 lookup(자동 추론 없음)·safe floor 출하 · reviewer-safe posture 는 구조적(read-only/approval-never/ignore-user-config)·tested-vectors-only 한정 · 산출 위치 canonical layout 한정) → spec 문장으로. per-key as-built enforcement 표·precedence 분기 세부·진단 호출 예시 → owner 가 이미 `config/reviewer.schema.json`(per-key description) + `scripts/review-run.ps1`(분기 구현) 이므로 spec 은 owner 명명으로 갈음하고 본문은 git history.
- **`docs/policies/REVIEW_EFFORT_GUIDE.md` — 흡수 후 retire.** durable 운영 원칙(effort ⟂ coverage · wired safe-default xhigh 에서 명확히 단순한 local-correctness packet 만 의식적 downgrade · 최소 의미 target set · 거대 무관 diff 금지 · verdict 별 처리)은 contract 흡수분과 중복되므로 spec 한 곳으로 합류. review-required/skip 분류·effort tier 표·checklist·examples 는 advisory 서사로 git history(향후 필요 시 별도 결정으로 deployed skill 측 보강 — 이 batch 비범위).
- **`docs/systems/review/STATUS.md` — 3분리 후 retire (기준 C).** maintenance-mode posture·LTS 종료·strict C1 canonical topology·operator entrypoint·verdict 비승인 → spec. Accepted residual risks 2건(operator narrow-scope over-confidence · bash chain final-exit-code false-fail)은 한 줄 + reopen 조건 형태가 성립하므로 → review_backlog rows. Completed ledger → git history; 단 ledger 행 RV-B-06/07 의 "global activation a separate pending action" 시점성 기재는 구현 단계에서 현재 사실(글로벌 설치본 상태)을 검증 후 처분한다 — stale 이면 기재 없이 종결, 사실이면 backlog row 로.
- **`docs/systems/review/BACKLOG.md` — `review_backlog.md` 로 통합 후 retire.** open RV-B-01/03/04/05 를 한 줄 + reopen 조건으로 이주(ID 보존). closed tombstone 3행(RV-B-06/07/08)은 row 를 만들지 않는다(closed = 행 삭제 원칙); tombstone 예외는 reference sweep 의 bare-token 기계 판정으로만. `next ID:` 는 기존 RV-B 계열의 단조 연속(RV-B-09)으로 Spec 에서 확정.
- **`docs/systems/review/IDEAS.md` — `review_backlog.md` 로 통합 후 retire.** idea-only 2건(devil's-advocate pre-pass · multi-reviewer consensus/calibration)을 각 한 줄 + reopen 조건(evidence threshold + maintenance-mode exit 별도 승인 + 별도 scoped goal)으로 이주하되 **idea-only 한정어를 행에 보존**한다 — implementation 후보가 아니라는 class 구분(RV-07 결정)은 reopen 조건의 일부다. "why not planned" 서사는 git history.
- **당시 lifecycle 기록 10종 — 전부 retire (기준 B).** `REVIEW_POLISHING_DECISION_RECORD.md` · `REVIEW_POLISHING_IMPLEMENTATION_PLAN.md` · `REVIEW_POLISHING_BATCH_A_SPEC.md` · `REVIEW_POLISHING_BATCH_D_SPEC.md` · `REVIEW_RESULT_PROVENANCE_SPEC.md` · `REVIEW_ARTIFACT_PERSPECTIVE_LAYOUT_PLAN.md` · `REVIEW_INPUT_KNOWN_CONCERNS_HYPOTHESIS_FORM_PLAN.md` · `REVIEW_LOCAL_VALIDATION_CLOSEOUT_POLICY_PLAN.md` · `REVIEW_RUNNER_STDOUT_ANCHOR_TEST_PLAN.md` · `REVIEW_VALIDATION_EVIDENCE_REPRODUCTION_POLICY_PLAN.md`. 전부 "구현 완료 + 현 상태 권위는 다른 곳" 을 자기 배너로 선언한 당시 기록이다. 단 retire 전 owner absorption proof 필수 — 특히 ① DECISION_RECORD 의 canonical terminology(dual-perspective coverage 5축 어휘)와 downstream 보존 invariant 목록, ② BATCH_D_SPEC 의 H1/H2/H3 세 home 분류(contract §6b 가 live pointer 로 참조 중), ③ PROVENANCE_SPEC·LOCAL_VALIDATION_PLAN 의 "policy origin" 지위 — 이들의 unique live 의미가 contract 경유 또는 직접으로 review_spec 에 표현됨을 Work Packet 의 line-level 분류로 입증한 뒤 삭제한다. 도메인 어휘(5축 등)의 home 은 review_spec(도메인 의미는 review owner surface 소유 — glossary 는 route-only)으로 한다.
- **glossary pending 2종 close (방향 결정; glossary mutation 은 별도 승인).** ① `evidence contract absorption` → 위 EVIDENCE_CONTRACT 처분으로 close — evidence contract 는 review 도메인(review_spec)이 흡수하고 evidence umbrella 는 rejected 유지. 신규 용어 채택 없음(pending 행 제거 + close 기록). ② `review-support naming` → **비채택 close** — repo 전체에서 이 토큰의 live 사용처가 glossary 자신뿐임을 검증했다(사용 0건); 명명 대상인 review 경로·artifact 의 canonical 이름은 layout·scripts 가 이미 소유한다(task/perspective/pass · input.md/result.md). 부재한 사용을 위한 우산 명칭을 만들지 않는다.
- **routing.** `docs/current/REPO_READING_GUIDE.md` 의 review 항목(Q2·Q3)은 `docs/README.md` 로 이관·제거(과도기 규칙: 새 orientation 은 README 만; 가이드 자체 retire 는 batch P 소관). `docs/contracts/README.md` 의 review/·evidence/ 행과 `docs/policies/README.md` 의 reviewer 정책 2행 제거, `docs/backlog/INDEX.md` 의 review 행 갱신, `docs/README.md` §5 의 contracts/systems 행과 review 도메인 행 반영. root `CLAUDE.md`/`AGENTS.md` Docs trigger map 의 Review 행 갱신(mirror-edit + parity test).
- **inbound pointer.** 구계열 retire 로 깨지는 경로 참조(README.md·tests/README.md·scripts·config·tests 의 비-behavior 참조 — Owner surface model 의 전수 목록 — 와 타 도메인/architecture/decisions 문서들, 약 40파일 규모)는 **pointer 수준으로만** 갱신한다 — 과거 기록 서사 안의 언급은 "당시/구(현 …)" 주석 규약, 소유 도메인 문서의 의미 재진술 정리는 그 도메인 batch 소관. `docs/contracts/global-invocation/SHARED_GLOBAL_INVOCATION_CONTRACT.md` 는 interface/pointer 정합 수준만(거취는 batch I). 빈 폴더(`docs/contracts/review/`·`docs/contracts/evidence/`)는 제거하되 `docs/contracts/`·`docs/policies/`·`docs/systems/` 계층 자체의 제거는 batch P 소관.
- **Work Packet(결정 — 필요; Plan 이 선언).** 목적 = ① 17파일의 절/행 단위 current-bearing vs narrative 분류 대조표(특히 contract 절→spec 문장 대응, 당시 기록 10종의 unique live 의미 유무 판정), ② inbound 참조 표면의 4-class 분류 지도, ③ `log/evidence/**` 언급 표면의 사용 성격 분류(경로 interface 사용 vs evidence 형식 의미 의존 — 위 분리 판정의 재확인 입력). 위치 `docs/review/review_work_packet.md`(committed temporary document — closeout 시 삭제, 보존 = git history). 흡수 대상 = review_spec(지속 결정만) + operator report. **승격 금지 조건**: Work Packet 의 어떤 내용도 live 문서로 복사·승격되지 않는다 — 지속 결정만 spec 의 normative 문장으로 재서술되어 들어간다.

## 하지 않을 것 (non-goals)

- review scripts / skill / templates / config / tests 의 **behavior 변경 없음**(maintenance-mode 유지). scripts/tests 의 주석 포인터 갱신은 비-behavior 한정이며 full Pester + verify-ps1 로 무변경을 입증한다. 판별 절차상 docs-stale 이면 docs 측 정정만 한다(구현이 판정 기준).
- `log/review/` 규약(three-level layout·2-file·write-once)·verdict 어휘(`yes`/`no`/`yes with risk`) 불변. 새 기능·reviewer adapter·multi-reviewer orchestration·parser gate 확대 없음 — IDEAS 항목의 승격 아님.
- evidence / global-invocation / instruction-surface 의 umbrella 도메인 부활 없음(glossary rejected 유지). `INSTALL.md` 불가침. `snippets/**` 글로벌 배포 티어 변경 없음(skill·snippets·templates 의 docs 경로 참조 0 은 전수 sweep 으로 확인된 사실이고, scripts·config·tests 의 비-behavior 참조 갱신은 Owner surface model 에 한정 명시된 scope 다).
- 글로벌 설치본(channel 3)의 refresh/update 는 이 batch 의 범위 밖(별도 명시 결정).

## Plan readiness / open risks

**Plan readiness: 내려갈 수 있다.** 구조물별 처분(구계열 17파일 + glossary 항목 2종)·owner 경계·흡수 고도·Work Packet 필요성이 결정으로 닫혔다. Plan 은 batch 내부 순서 / hard boundary / validation expectation / review focus / Work Packet 선언과 아래 open 의 close 지점을 확정한다.

**Open decisions (close 예정지 포함):**

1. **contract 절별 흡수/소거의 line-level 대응** — 어느 절이 spec 문장이 되고 어느 절이 active-surface owner 명명으로 갈음되는가. close: Work Packet 작성 후 Spec 에서(누락·중복 0 대조).
2. **STATUS ledger 의 "global activation pending" 기재의 현재 사실 여부** — close: 구현 단계에서 글로벌 설치본 대조 후(stale 이면 종결, 사실이면 backlog row).
3. **residual risk 2건의 backlog 문안**(reopen 조건의 한 줄 압축) — close: Spec/구현 단계의 backlog 작성에서.
4. **`docs/README.md` 혼합 상태 문안** — close: closeout Level-1 gate 에서.
5. **(상시 조건) 모델-결함 신호** — review_spec 이 회차성 내용을 요구하기 시작하거나 형틀이 이 도메인을 표현하지 못하면 batch 중단 → 외부 Design rewind. close 지점 없음(상시 감시).

**Open risks:** ① 이 도메인의 흡수 규모가 지금까지 중 최대다(contract 단독 600줄대) — spec 이 분량 압력으로 prose mirror 화되거나 반대로 과소 흡수로 unique live 의미를 떨어뜨릴 양방향 risk 가 있다; Work Packet 의 절 단위 분류와 Spec review focus("목표 상태 명세인가")가 통제 지점이다. ② 당시 기록 10종의 owner absorption proof 누락이 이 batch 의 최빈 실패 모드다(잔재 최다 도메인) — 파일별 unique-live-meaning 판정을 Work Packet 에서 명시 대조한다. ③ inbound 참조 표면이 넓다(약 40파일) — 4-class sweep(삭제 변경의 granular technique 포함) + corrected-state review 로 닫는다. ④ scripts·config·tests 의 비-behavior 참조 갱신(주석·schema description·문서 포인터)이 behavior 불변과 양립함은 full Pester + verify-ps1 로 입증하며, 이 갱신은 글로벌 설치본과의 한시적 wording 편차를 만든다(설치본 refresh 는 별도 결정; review engine 은 global stable 이라 무영향). ⑤ 비례성 출구 사용 횟수는 closeout 보고에 한 줄로 센다.
