# install-update Design

## Header

**이 문서는 무엇의 Design 인가.** install-update 도메인 마이그레이션(batch I)의 방향 문서다 — 분산된 install-update 문서 표면(`docs/systems/install-update/` 7파일 · `docs/contracts/global-invocation/SHARED_GLOBAL_INVOCATION_CONTRACT.md`)을 docs-working-model 의 도메인 구조(`docs/install-update/`)로 수렴시키는 변경의 why / what / owner-surface 결정을 담는다.

**이 체인이 끝나면 무엇이 되는가.** `docs/install-update/install-update_spec.md`(목표 상태 명세, 구현과 1:1 동기화)와 `install-update_backlog.md` 가 live 가 되고, 구계열 문서·routing 항목은 retire 되어 install-update 도메인이 자기 폴더 안에서 닫힌다. global-invocation contract 거취와 glossary pending 2종(`run diagnostics` · `global-invocation single-home`)이 close 된다.

**이 문서가 아닌 것.** Plan 아님 · Spec 아님 · Work Packet 아님 · install/update/uninstall/activation scripts·config·tests 의 behavior 변경 아님 · `INSTALL.md` 의 재작성 아님. 이 Design 은 mutation/commit/push 승인이 아니다(1회 진술).

## 왜 바꾸는가 / 무엇을 바꾸는가

**왜.** install-update 도메인의 normative 사실이 네 표면(모델 문서 1 · 결정 기록 가이드 1 · uninstall 설계 1 · cross-domain contract 1)과 상태 4종(STATUS/BACKLOG/DEFERRED/IDEAS)에 분산되어 있고, 그 형태가 모델의 다섯 산출물 분류와 비정합하다. `GLOBAL_INSTALL_UPDATE_MODEL.md`(617줄)는 스스로 "모델/설계 source record(실행 authority 아님 — 실행은 `INSTALL.md`)"를 선언하면서 1차/2차/3차 BRIEF reconciliation·source-cache canonicalization supersede 등 **층층의 historical framing 을 본문에 보존**해, 현재 의미를 읽으려면 note 들의 우선순위 규칙을 따라가야 하는 상태다 — current-bearing 의미와 historical narrative 가 한 파일에 얽혀 있다. `STEP3_INSTALL_UPDATE_DECISION_GUIDE.md`(1,359줄)는 "Recorded … decision anchor" 형식의 당시 결정 기록 모음이고, `UNINSTALL_LIFECYCLE_DESIGN.md`(259줄)는 design+as-built+incident 기록 혼합이며, contract(409줄)는 as-built 구현 후 specification-of-record 역할을 하고 있어 자리만 다른 spec-of-record 가 두 개 생길 수 있는 상태다. 미이주 도메인 중 분량 최대(대형 4문서 합계 2,644줄). single-home 원칙 하에서 이 사실들의 단일 거처는 도메인 폴더의 spec/backlog 다.

**무엇을.** ① `docs/install-update/` 도메인 폴더에서 install-update_spec 을 생산하고(아래 처분 결정에 따라 구계열의 current-bearing 내용을 normative 문장으로 재구성), ② BACKLOG·DEFERRED·IDEAS·STATUS 잔여 risk 를 `install-update_backlog.md` 로 통합하고, ③ 구계열 문서를 retire 하고, ④ routing 을 `docs/README.md` 로 이관하고, ⑤ glossary pending 2종을 close 하고, ⑥ 배포 표면 주석 3건을 self-contained 화하고, ⑦ `INSTALL.md` 의 stale 화되는 pointer 2곳을 최소 정정한다(사용자 합의 — 의미 불변·pointer-only). 구현 behavior 변경은 0 — 이 변경은 문서 표면의 수렴이다.

## Owner surface model

- **behavior owner 는 변하지 않는다**: `scripts/install-global.ps1` / `update-global.ps1` / `install-update.ps1` / `uninstall-global.ps1` / `uninstall-finalizer.ps1` / `activate-global.ps1` / `apply-managed-block.ps1` + `scripts/lib/`(`install-pipeline-core.ps1` · `activation-surface.ps1` · `managed-block.ps1` · `uninstall-target.ps1` · `path.ps1` · `resolve-script.ps1`)가 lifecycle behavior·status 어휘 emission·ToolRoot/ProjectRoot resolution 의, `tests/`(install/update/uninstall/activation/path/resolve-script suites)가 지속 검증의 owner 다. **`INSTALL.md` 는 install/update/reinstall/uninstall 실행의 유일한 operative contract 다**(self-contained·anti-coupling — 실행 시 `docs/**` 를 읽지 않는다). install-update_spec 은 이들을 명세하고 대조될 뿐 operative authority 가 아니다(root *Final hard rule*).
- **spec 과 `INSTALL.md` 의 점유 경계(결정 — 이 도메인 고유).** 이 도메인은 operative 실행 계약(`INSTALL.md`)이 의도적 contextual duplication 으로 self-contained 한 유일한 도메인이다. spec 은 `INSTALL.md` 를 대체·요약·재진술하지 않는다 — spec 의 점유는 **도메인의 목표 상태·소유 경계·durable invariant**(layer model 의 계층 경계, footprint contract, reinstall-first posture, activation-surface 정책, 승인 경계 분리)이고, `INSTALL.md` 의 점유는 **실행 절차·구체 값**(명령·경로·필드·status 문자열·단계 순서)이다. 충돌 시 실행은 `INSTALL.md` 가 결정한다(이 우선 관계 자체를 spec 에 명문화한다). 둘 사이 중복 0 은 Work Packet 의 점유 경계 대조로 입증한다.
- **흡수 고도(결정).** spec 의 1:1 단위는 normative 문장이다 — 외부 관찰 가능 행동·소유 경계·standing boundary 를 명세하고, 기계 세부(install metadata 14-field schema 값·manifest/marker 필드·status 문자열 enumeration·channel 별 구체 경로 값)는 active surface(`INSTALL.md`·scripts·schema·snippet)가 소유하는 값으로 두고 spec 은 그 invariant 와 owner 만 명명한다. 대형 문서의 절 구조를 형식째 복제하지 않는다(spec 은 prose mirror 가 아니다).
- **배포·active 표면의 참조 실태와 경계(결정).** 배포 payload 중 snippets·skills·templates·config 와 tests 는 install-update 도메인 docs 경로를 참조하지 않는다(대상 파일명 전수 sweep 은 구현 단계에서 재확인). 반면 scripts 는 retire 대상 경로의 **비-behavior 주석 참조 3건**을 가진다: `scripts/lib/activation-surface.ps1:11`(MODEL §8A rationale) · `scripts/lib/managed-block.ps1:233`(UNINSTALL §4) · `scripts/lib/install-pipeline-core.ps1:40`(STEP3 §12). 이들 주석의 self-contained 갱신(직접 결정 문장화 — 도메인 docs 경로/기록-의미 참조 신설 금지, batch R 에서 확립된 배포-표면 invariant 와 동일)을 이 batch 의 scope 에 포함한다 — 전부 비-behavior 이며, 무행동변경은 full Pester + verify-ps1 로 입증한다. scripts 는 배포 payload 이므로 이 갱신은 repo 와 글로벌 설치본 사이에 한시적 비-behavior wording 편차를 만든다(설치본 refresh 는 별도 명시 결정; 운용 무영향).

## 수정 대상 (구조물별 처분 결정)

수정하는 기존 live Spec 은 없다(이 도메인의 첫 마이그레이션). 처분 판정 기준은 batch R 의 기준을 재사용한다: **(A) consumer 판정** — 외부 adopter 가 구현 없이 준수해야 하는 입출력 계약인가(yes → 별도 contract role 존속, no → spec 흡수); **(B) current-bearing 판정** — 당시 기록 중 unique live 의미만 흡수하고 나머지는 git history; **(C) 3분리** — posture/capability → spec, open 항목 → backlog, narrative → git history.

- **`GLOBAL_INSTALL_UPDATE_MODEL.md` — 분리 흡수 후 retire (기준 B+C).** current-bearing 으로 spec 에 흡수: 5-layer model 의 계층 경계와 각 layer 의 소유 의미 · source 획득 2-mode 의 공통 구조와 목표 방식 결정 · reinstall-first recovery posture(+activation surface 가 generated payload 와 다른 별도 영역이라는 2분류 예외 — managed-block 은 marker-bounded replace, skill mirror 는 canonical-overwrite) · update 의 metadata-dispatch 의미와 두 사용자 명령의 의미 경계 · install metadata 의 위치/성격 invariant(instance 는 global layer only·target 생성 금지·schema/example 만 source 에) · **target footprint contract(`<ProjectRoot>/log/` only — 타 도메인이 interface 로 참조하는 경계의 home)** · **§8A deployed runtime extension activation-surface 정책 전체**(STATUS 가 single home 으로 지정한 governance — forced-copy+final-verification·generic inventory model·신규 extension checklist) · self-adoption 결정(source payload vs project-local state 분리·검증의 의미·separate state root not-default) · §12 non-goals 중 standing 금지 경계(transaction/rollback framework 미도입·auto daemon 금지·installer-first productization 금지·automation 본체와 managed-block apply 의 승인 scope 분리). historical 로 git history 보존: 1차/2차/3차 BRIEF reconciliation lineage·source-cache canonicalization superseded framing·§10 diagrams·§11 관계 서사·§13 execution status note. 절→문장 대응의 line-level 대조는 Work Packet 소관 — 특히 이 문서는 superseded wording 이 본문에 의도적으로 보존된 형태라 **current 의미의 추출은 반드시 note 우선순위 규칙(3차 기준)을 따라 분류**한다.
- **`STEP3_INSTALL_UPDATE_DECISION_GUIDE.md` — current-bearing 흡수 후 retire (기준 B).** "Recorded … decision anchor" 모음(§10~§19)이며 자기 역할을 Step 3 구현 계획 안내(MODEL 의 subordinate)로 선언한다. 구현이 완료되고 operative 계약이 `INSTALL.md` 와 scripts 로 닫힌 현재, anchor 의 결정 중 owner 가 이미 소유한 것(pipeline 계약 → scripts/tests·`INSTALL.md`, manifest/marker 필드 → schema·`INSTALL.md`)은 spec 의 owner 명명으로 갈음하고, durable 경계로 spec 문장이 필요한 것 — source-cut "deferred with exact boundary"(detection-only) · dogfooding tree-separation invariants 와 `-AllowDogfoodSource` 경계 · fixture-only CLI 경계(production global apply 구조적 차단) · §19 operating policy(no automatic recovery·AI-guided apply) · 각 anchor 의 "작업 후보 아님" 중 standing 금지 — 를 Work Packet 분류로 선별 흡수한다. 결정 경위·cross-review 서사·lineage(§1~§9)·closeout 기록(§13)은 git history.
- **`UNINSTALL_LIFECYCLE_DESIGN.md` — 흡수 후 retire (기준 B).** durable 로 spec 에: 성공 기준(global footprint zero) · temp finalizer trampoline invariants · managed-block 은 file 삭제가 아니라 marker-span excision · install-root 제거는 expected-footprint enumeration + unexpected-content fail-fast · non-targets 목록 · dry-run/apply/verify 분리 · failure policy 2-tier(preflight-all-then-act vs per-surface partial) · status 어휘의 standalone 성격(어휘 값 자체의 owner 는 scripts·`INSTALL.md` §11(b) 기재). as-run dogfood(§13)·main-PC incident(§14) 서사는 git history(IU-B-10 의 current 의미는 docs hardening 완료 사실로 ledger 와 함께 종결).
- **`SHARED_GLOBAL_INVOCATION_CONTRACT.md` — 흡수 후 retire (기준 A: 별도 contract role 불채택 — 방향 판정; Work Packet 의 unique-live-meaning 검증 후 Spec 에서 확정).** 판정 근거: ① 이 toolset 의 배포는 docs-free 이고 배포 표면(snippets·skills·templates·config·scripts)은 이 contract 경로를 참조하지 않는다 — adopter 가 만나는 invocation topology(channel chain)는 deployed snippet 이 self-contained 로 운반하고 `scripts/lib/path.ps1`(+`resolve-script.ps1`)가 구현한다; "구현 없이 준수" 시나리오가 존재하지 않는다. ② 문서 스스로 D1–D9 결정의 as-built 기록 + formalized contract 임을 선언한다 — 구현 완료 후 그 역할은 도메인 spec 과 동일하므로 공존은 single-home 위반이다. ③ 외부 참조처(root README·`docs/project/AI_HARNESS_TOOLSET_SCOPE.md`·`docs/roadmap/INDEX.md`·architecture 기록류)는 전부 orientation/pointer 사용이다. 흡수 내용: channel chain invariant(우선순위·channel 3 default·channel 2 의 override 용도 경계·channel 4 dogfooding 한정·channel 5 legacy) · ProjectRoot 해석(D9 advisory 포함) · cross-tree write isolation(runtime 산출은 ProjectRoot `log/` 로만) · self-target enforcement(D8 — live)의 **interface 진술**(그 의미의 owner 는 verify-ps1 — 재진술 금지) · 구현 분할 지도(§6)는 owner 명명으로 갈음. **단 D6(review-verify 의 sidecar `toolRoot` binding)·D7(untracked exclusion)은 wording 이 아니라 mechanism 자체가 현행 active surface 에서 제거된 superseded 항목이다(각 섹션 머리 Superseded note) — 흡수 대상이 아니며 git history 보존, spec 문장으로 부활 금지.** 그 외 superseded review-cycle wording(§2 inputs·§5.4·§5.5·§6 일부 — 문서 상단 supersede note 적용 범위)도 동일하게 historical 분류다 — live vs superseded 경계의 line-level 대조는 Work Packet 소관. 거취가 흡수로 확정되면 `docs/contracts/global-invocation/` 빈 폴더를 제거한다(`docs/contracts/` 계층 자체는 batch P 소관).
- **`STATUS.md` — 3분리 후 retire (기준 C).** posture → spec: lifecycle subsystem 의 LTS maintenance 상태 · entrypoint split 이 intended architecture 라는 결정(merged mega-CLI 아님) · default adoption shape(channel 3 stable runtime ToolRoot·legacy channel 5 는 호환 전용) · recovery=reinstall-first·integrity=manifest+marker · change discipline(scoped goal+review gate·installer-framework class 확장 금지) · Non-claims 의 승인 경계(한 환경 dogfood 가 다른 환경 apply 를 승인하지 않음 등). Completed ledger(IU-01..15)·Operational closeout ledger(IU-OPS-01..05) → git history. Accepted residual risks → backlog rows(reopen 조건 보존 — IU-B-07 RETIRED 경계·per-environment 승인 경계·D8 narrowed residual 포함; 한 줄 성립 여부는 Spec/구현 단계 문안에서).
- **`BACKLOG.md` — `install-update_backlog.md` 로 통합 후 retire.** open IU-B-01..06·IU-B-13 을 한 줄 + reopen/start 조건으로 이주(ID 보존). closed tombstone 5행(IU-B-07/08/09/10/12)은 row 를 만들지 않는다(closed = 행 삭제 원칙; 예외는 reference sweep 의 bare-token 기계 판정으로만 — 특히 IU-B-07 은 IDEAS 이주 row 가 RETIRED 한정어로 그 경계를 운반하므로 tombstone 불요 예상). `next ID: IU-B-14` 로 단조 연속.
- **`DEFERRED.md` — backlog 통합 후 retire.** IU-D-01/02/03/07/08 을 reopen 조건 보존 한 줄로 이주(deferred class 한정어 유지 — backlog open 후보와 결정-연기를 구분). STATUS 가 언급한 `-AcquisitionClonePath`/URL-normalization 결정의 현재 행방은 Work Packet 에서 확인 후 row 화 또는 종결. "Not deferred — explicitly not work candidates"(STEP3 §15.6 out-of-scope 입장)는 standing 금지 경계로 spec durable boundary 에 표현한다(재개 = 그 입장을 먼저 reopen 하는 별도 결정).
- **`IDEAS.md` — backlog 통합 후 retire.** retired 항목 1건(one-shot NL update / activation auto-apply)을 idea-only + RETIRED 한정어 한 줄 row 로 이주 — 4단계 reopen 경로(별도 reopen 결정+row re-entry+scoped goal+review gate)가 reopen 조건이다. safe-condition outline 서사는 git history.
- **glossary pending 2종 close (방향 결정; glossary mutation 은 구현 단계).** ① `global-invocation single-home` → contract 흡수 판정으로 close — global-invocation contract 의 single home 은 `install-update_spec.md`(구현 owner = `scripts/lib/path.ps1`+snippet topology). broad "global-invocation 도메인" 은 rejected 유지·부활 없음. ② `run diagnostics` → **비채택 close** — repo 전체에서 이 토큰의 live 사용처가 glossary 자신뿐임을 검증했다(사용 0건); observed-run-state/status 어휘의 owner 는 `INSTALL.md` 의 fixed-vocabulary status(§13)와 scripts emission 이 이미 소유한다. 부재한 사용을 위한 명칭을 만들지 않는다(`INSTALL.md` operative wording 불가침과 정합 — blanket-clean 없음).

## Plan readiness / open risks

**Plan readiness: 내려갈 수 있다.** 구조물별 처분(구계열 7파일 + contract 1 + glossary 2종)·spec↔`INSTALL.md` 점유 경계·owner 경계·흡수 고도·Work Packet 필요성이 결정으로 닫혔다. Plan 은 batch 내부 순서 / hard boundary / validation expectation / review focus / Work Packet 선언과 아래 open 의 close 지점을 확정한다.

**Open decisions (close 예정지 포함):**

1. **contract 거취의 최종 확정** — 방향은 흡수(기준 A 불충족)이나, D1–D9·§5 에 별도 contract role 을 요구하는 unique live meaning 이 있는지 Work Packet 분류로 검증 후 Spec 에서 확정.
2. **대형 문서 절별 흡수/소거의 line-level 대응** — 어느 절이 spec 문장이 되고 어느 절이 owner 명명으로 갈음되는가. close: Work Packet 작성 후 Spec 에서(누락·중복 0 대조).
3. **uninstall status 어휘 등 standalone 어휘의 spec 문장 수위** — 어휘 값 enumeration 은 owner(scripts·`INSTALL.md`)에 두고 spec 은 invariant 만 명명하는 선을 Spec 에서 확정.
4. **residual risk·deferred 의 backlog 문안**(reopen 조건의 한 줄 압축 + deferred 한정어) — close: Spec/구현 단계의 backlog 작성에서.

**Open risks:** ① **분량 최대 + 1:1 문장화 분량의 첫 측정점** — 대형 4문서 합계 2,644줄; spec 이 prose mirror 화되거나 과소 흡수로 unique live 의미를 떨어뜨릴 양방향 risk. 통제 = Work Packet 의 절 단위 분류 + spec 작성 전 사전 축 합의 + Spec review focus("목표 상태 명세인가"). ② **`INSTALL.md` 인접성** — spec 이 operative 영역을 재진술하면 single-home 위반이자 duplication 신설; Work Packet 의 점유 경계 대조(③축)가 통제 지점이다. ③ **interface vs semantics 판정** — 이 도메인은 cross-domain 메커니즘이라 예외적으로 타 도메인 접점이 많지만(D6 review-verify·D7 untracked·D8 self-target·footprint interface 의 소비자들) 허용은 interface 진술까지다 — 타 도메인 semantics 재진술이 가장 빠지기 쉬운 함정. ④ inbound 참조 표면이 넓다(약 34파일/192건) — 4-class sweep(retire 대상 전체 파일명 기준 case-insensitive 전수) + corrected-state review 로 닫는다. ⑤ scripts 주석 갱신이 글로벌 설치본과 한시적 wording 편차를 만든다(설치본 refresh 별도 결정; install/review 운용 무영향). ⑥ 비례성 출구 사용 횟수는 closeout 보고에 한 줄로 센다.
