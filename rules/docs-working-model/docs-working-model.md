# Rule: Documentation working model (repo-only)

Applies to developing the `ai-harness-toolset` repository — the binding rules for the repo's **document artifact classes**, the **Design → Plan → Spec lifecycle**, **docs placement (end-state + transition)**, **Spec ↔ implementation synchronization**, and the **closeout gate**. This rule is self-contained: it does not delegate its meaning to any `docs/**` page. Read this rule **before** changing `docs/` content, producing a lifecycle document, or closing out work; the root `CLAUDE.md` / `AGENTS.md` *Docs trigger map* (`Source / docs` row) wires that trigger.

> **Package note.** This file is the operative home of the `rules/docs-working-model/` rule package. The package carries `templates/` (Design / Plan / Spec — `templates/docs-working-model_design_template.md` / `_plan_template.md` / `_spec_template.md`) and `checklists/` (Design / Plan / Spec / Work Packet conformance + closeout + promotion-boundary / promoted-but-not-live — `checklists/docs-working-model_design_checklist.md` / `_plan_checklist.md` / `_spec_checklist.md` / `_work_packet_checklist.md` / `_closeout_checklist.md` / `_promotion_checklist.md`). Package entry / routing is via `rules/README.md`; this operative home routes to the package-internal forms (see *Template / checklist conformance gate*).

## When this rule applies

- Any task that **places, moves, changes, or closes out** `docs/` content.
- Any task that produces or revises a repo document through the **Design → Plan → Spec lifecycle** — authoring a Design / Plan / Spec, updating a live Spec, closing one out, or retiring an absorbed Design / Plan.
- Any creation or disposal of a temporary work artifact (Work Packet or `_incubation` document) or a future-work queue entry.

## Document artifact classes (five)

Every repo document artifact belongs to exactly one class; mixing roles across classes is a defect:

1. **Planning artifacts** — Design / Plan / Spec (lifecycle below). Only the Spec stays live after closeout.
2. **Temporary work artifacts** — **committed temporary documents** carried by git until closeout, then deleted (preservation = git history): the **Work Packet** (round-scoped; see *Work Packet*) and, for a pre-promotion candidate (a domain **or** rule candidate), the **`_incubation` document** (candidate-lifecycle-scoped, not round-scoped; disposed at its **candidate-lifecycle closeout** — the candidate's promotion or discard, which in the promotion path precedes the promoted-lifecycle closeout; see *Incubation tier*).
3. **Operator reports / closeout evidence** — execution results, review results, validation evidence, point-in-time states. Live under `<ProjectRoot>/log/**` (runtime area), never in planning-artifact bodies.
4. **Active implementation surfaces** — scripts, skills, snippets, templates, config, tests, root instruction files, `rules/**`. The active surface owns behavior; a doc describes it (root *Final hard rule*).
5. **Future-work queue** — per-domain `<domain>_backlog.md` (see *Future-work queue*).

## End-state placement and transition

**End-state (declared now; executed only per-domain).** The `docs/` tree converges to domain folders directly under `docs/`:

```text
docs/
  README.md                      # the only orientation map (placement + minimal routing)
  <domain>/
    <domain>_spec.md             # live — the domain's spec-of-record
    <domain>_backlog.md          # compact future-work queue
    <domain>_design.md           # exists only during a change (retired at closeout)
    <domain>_plan.md             # exists only during a change (retired at closeout)
    <domain>_work_packet.md      # exists only during a change (deleted at closeout)
```

- **Execution is per-domain-batch only.** This declaration creates no immediate mutation; each legacy structure migrates in its own scoped batch with owner absorption proof + reference sweep + review gate.
- **End-state wins conflicts.** When a new decision conflicts with a legacy structure, decide toward the end-state; execute in that domain's batch.
- **Legacy structures persist but do not grow.** Do not add new narrative/items to retirement-bound structures (STATUS narratives, decision bodies, architecture residue); unavoidable corrections stay at proportionality-rule level.
- **Transition orientation.** During the mixed period, **new** orientation/routing content lands in `docs/README.md` only — it is the single orientation source for the transition and the Level-1 gate surface. The per-question read-first routing has since converged into `docs/README.md` §7 (the former `docs/current/REPO_READING_GUIDE.md` routing surface has been retired). Any legacy routing surface still awaiting its migration batch remains unmigrated residue: it keeps serving its **existing** read-first routing until that batch, but no new routing content is added to it (it is not a home for *new* routing). Each domain batch's closeout verifies `docs/README.md` reflects the mixed state (see *Closeout — reduced two-level gate*).

**single-home-plus-pointers.** Every fact has exactly one authoritative home; all other mentions are pointers to it, never copies. Duplication is the staleness engine: a duplicated fact needs an N-place sweep on every change, and any missed place silently goes stale.

**Durable-pointer prohibition.** A committed doc must never use a durable pointer to a gitignored / local / scratch / runtime path (`log/**`, `polishing/**`, `repo_snapshot/**`, repo-sibling artifacts, runtime evidence, user/global files); durable pointers resolve only to git-tracked files or git commit/history. Externalized historical detail is preserved by git history — if a past decision is still operationally relevant, the active doc states the decision directly.

## Design / Plan / Spec lifecycle

A repo document / normative change is produced through a fixed lifecycle, not authored ad hoc:

```text
live Spec + live Implementation → 변경 필요 → Design → Plan → live Spec 을 목표 상태로 갱신
  → (필요 시 Work Packet) → Implementation (final Spec only) → closeout sync
  → Design/Plan/Work Packet 의 current-bearing 내용 흡수 → Design/Plan retire · Work Packet 삭제
  → Spec live · Implementation live
```

**Lifecycle invariant — detail flows downward (altitude per artifact).** Within the **Design → Plan → Spec / terminal-rule lifecycle**, each canonical artifact holds only its own altitude, and detail flows to its owning artifact **by kind**, never bucketed upward: direction rationale / conceptual model / chosen trade-offs (incl. decision-grade rejected alternatives) / ownership boundary / non-goal scope / the semantic target → **Design**; approval-target decisions → **Plan**; round-scoped investigation / line-level analysis / implementation notes → **Work Packet**; the durable target-state *specification* / normative text → the live **Spec**, or for a `rule_docs` item the **terminal rule file** (a rule is its own spec-of-record — no separate Spec). Front-loading a lower-altitude detail into a higher artifact is a defect (it over-commits before the deciding artifact); pushing an approval-target decision *down* into a Work Packet is the inverse defect (it escapes the approval gate). Omitting a lifecycle artifact does not promote its content to another home — create the owning artifact, or drop / externalize the detail; never smuggle it upward. The **other artifact classes keep their own homes** (owned in their own sections, never re-routed by this invariant): execution records / outcomes → operator reports under `log/**`; future-work items (reopen / start condition, `next ID`) → the `<domain>_backlog.md`; a term's one-line meaning / classification → `rules/terminology-glossary.md` (full semantics → its finalization-owner); a behavior's final wording → the **active surface** that owns it. **Special paths keep their own model:** the pre-promotion `_incubation` document is the candidate's single, deliberately multi-grade planning home (*Incubation tier* governs it, not this invariant); a carried-over prior-revision artifact is triaged under *State migration* (non-authoritative until judged, not an automatic defect). This invariant binds **authored, current-bearing** lifecycle content, so a meaning-preserving direct edit (per the *Proportionality rule*) may use uncommitted / conversational scratch rather than a committed artifact.

- **Design** — why / what / owner-surface model / chosen trade-offs / non-goals / which live Spec or implementation it modifies; a **decision-grade direction** artifact, not permanently live. A Design carries the *semantic target* — a paraphrasable statement of what the change must come to mean — plus the decision-grade content needed to align direction: conceptual model, ownership boundary, the deciding target-state invariant, decision-critical identifiers / interface names, a closed semantic enum or taxonomy *when that taxonomy is itself the decision*, and representative / boundary examples. It does **not** carry lower-grade detail — round-scoped / line-level analysis, execution sequences / staging / mechanics, exhaustive candidate / implementation inventories, or the final normative wording (the MUST / forbid sentence form) — which flow to their own homes per the *Lifecycle invariant* above. This is the Design's content boundary, parallel to the Plan's *not-a-work-memo* and the Spec's *must-not-contain* boundaries: the *semantic target* and decision-critical identifiers live in the Design at direction grade, while the durable target-state specification and the final normative wording are owned by the Spec / terminal rule.
- **Plan** — **approval-target decisions only**: batch order / per-batch scope / hard boundaries / validation expectations / review focus / Work Packet declarations (purpose, absorption target, retire condition) / open-decision close points. A Plan is not a work memo — investigation results and round-scoped analysis belong in a Work Packet; execution sequences and execution records belong in operator reports under `log/**`.
- **Spec** — the domain's **target-state specification** (identity below). Live after closeout.
- **Implementation** — built from the **final Spec only** (no direct Design/Plan reference); it may consult a Work Packet, which never substitutes for the Spec. At closeout the Spec and the implementation are reconciled 1:1.

## Spec identity — target state + durable boundary

A Spec's meaning is time-phased: at writing completion it is the **blueprint of the target state to implement**; during implementation it is the implementer's reference; after closeout it is the **live specification 1:1-synchronized with the implementation**. Its *Lifecycle state* section names which point it is at: **`prelive`** (this Spec has been written — at the Spec stage of the promoted lifecycle, after the promotion transition's `_design` and the Plan — but closeout has not yet made it live; the writing-completion blueprint above), **`sync-required`** (a previously-live Spec updated in place, awaiting re-sync — see *Live-Spec update*), or **`live`** (closeout-reconciled 1:1). A `prelive` Spec is a **governance-discoverable target-state blueprint, not yet implementation authority** (E1). `prelive` is a **domain-Spec** state; a **rule candidate** has no Spec, so its promoted-but-not-live phase is simply its `_design` / `_plan` *active lifecycle work* (the 3-state snapshot) — it is **not yet a discoverable rule** until the terminal rule file lands, and so needs no separate marker.

A Spec carries (the spec template fixes these as its eight sections): **Header** (what this document is / chain outcome / what it is not — each within 3 lines) · **목표 상태** (what the domain is/must be, in normative sentences) · **Owner surface 지도** · **Durable boundary** (standing allow/forbid boundaries; rules and specs own the *class/invariant*, concrete path values are owned by the active surface / `INSTALL.md`) · **Cross-domain interface** (interfaces only, never another domain's semantics) · **Validation expectation** · **Review focus** · **Lifecycle state**.

**A Spec must not contain:** round-scoped candidate-file lists, execution command sequences, staging procedures, review results, readiness judgments, or point-in-time work status (the lifecycle-state section's compact markers excepted). Those compact markers are limited to **state markers** (e.g. live / sync-required, capability maturity, lifecycle-doc presence) and must **not** copy a backlog's item ID enumeration or its next-ID allotment — those are owned by the backlog as their single home and are referenced by pointer only (per *single-home-plus-pointers*). Round-scoped **analysis** (candidate-file classification, investigation notes) belongs to the Work Packet; execution mechanics and records (command sequences, staging procedures, review / validation results, readiness judgments, point-in-time status) belong to operator reports under `log/**` (the *Work Packet* content boundary below).

## Work Packet (temporary work artifact)

- **Role**: a round-scoped temporary work document — line-level reference classification, investigation notes, implementation notes, evidence proposals, reviewer-question preparation, edge-case notes. **Not an approval target, not a live domain document, never a substitute for the Spec.**
- **Content boundary (forbidden in a Work Packet)**: execution command sequences, staging procedures, review results, validation results, readiness judgments — these are execution mechanics / records and belong to operator reports under `<ProjectRoot>/log/**` (or are not recorded at all).
- **Location**: `docs/<domain>/<domain>_work_packet.md` for a domain's round work, or `rule_docs/<id>/<id>_work_packet.md` for a rule's round work (the per-rule planning workspace — see *Incubation tier > `rule_docs/`*) — a **committed temporary document** in the domain / rule folder, carried by git until closeout. No subfolder evasion (`docs/<domain>/work/` and `rule_docs/<id>/work/` are forbidden; the *Stable filename rule* applies). Being tracked makes it transferable, not live or authoritative.
- **Lifecycle**: created only when needed (a Plan may declare its necessity, absorption target, and retire condition); at its closeout — a work packet held during candidate incubation closes at the **candidate-lifecycle closeout**, an active-lifecycle work packet at the **promoted-lifecycle closeout** (see *Incubation tier*) — its current-bearing content is absorbed into the Spec / the correct owner surface / the closeout report (on a candidate **discard** there is no canonical-surface absorption — the discard commit message preserves the rationale); then the file is **deleted** — preservation is git history, like a Design/Plan retire.
- Boundary aid (when the Spec/Work-Packet line wavers): "is it still true after this round ends?" — true → Spec; false → Work Packet.

## Incubation tier (pre-promotion candidate stage)

A **candidate** — a capability or operating discipline under evaluation for whether it should become a promoted **domain** or a promoted **rule** — develops in a governed pre-promotion stage: in-repo but non-authoritative, neither an out-of-repo sprawl nor a premature domain/rule.

- **Admission.** A candidate enters incubation only with: a specific problem an existing domain/rule cannot cover, a candidate shape (expected single-home, not a bucket, the success-absorption artifact and the failure-deletion target named), an owner, a review-date, and discard criteria. Missing any → it stays out-of-repo scratch.
- **Incubation document.** One committed-temporary `<candidate>_incubation.md` per candidate, located by candidate kind: a **domain candidate** at `docs/<candidate>/`, a **rule candidate** at `rule_docs/<candidate>/` (the in-repo rule planning workspace — see the `rule_docs/` purity bullet below; **1:1 rule-bound** to a rule or candidate whose terminal output is a rule file, never a broad operating-philosophy / project-architecture / branching-strategy / domain bucket — that mixed-owner shape stays the rejected `docs/architecture/` shape under a new name). It is candidate-local (a class-2 lifecycle role, deleted at promote or discard) and the candidate's **single, self-sufficient planning home**: a session or subagent can start work from this document alone — no separate seed document is required. The incubation tier produces **no separate `_design` / `_plan` / `_spec` file**; canonical role *filenames* begin only at promotion (live *authority* begins later — at the live Spec for a domain, the terminal rule landing for a rule): the promotion transition writes the **entry promoted artifact (the `_design.md`)** for either kind, after which a **rule candidate** proceeds Design → Plan → rule (the terminal rule file is that lifecycle's eventual output — **no separate Spec**, a rule is its own spec-of-record) while a **domain candidate** proceeds the full Design → Plan → Spec lifecycle. Header carries `non-authoritative` / `not referenced by canonical rules/indexes` / `owner` / `review-date` / `open questions`. The `_incubation` role joins the *Stable filename rule* as a committed-temporary candidate lifecycle role, not a canonical domain/rule identity.
- **`rule_docs/` — the in-repo rule add/revise planning workspace (per-rule, persistent).** `rule_docs/` is the in-repo home for planning the *addition or revision of a rule* — kept in-repo (not out-of-repo scratch) so the work is tracked and managed. Placement mirrors `docs/<domain>/` in the *planning-doc pattern* (`<base>/<id>/<id>_<role>.md`, base = category: `docs/` for domains, `rule_docs/` for rules); the lifecycle stage lives in the filename suffix, never in the base folder name. `rule_docs/` is strictly **1:1 rule-bound** (each child maps to exactly one existing rule or one rule candidate, by id) and is **never** a general policy / architecture / philosophy bucket (that broad-bucket shape stays rejected). The output stays in the rule file itself (`rules/<rule>/<rule>.md`, or `snippets/rules/<rule>.md` for a global-distribution rule) — a rule is its own spec-of-record, so there is **no spec document in `rule_docs/`** and a rule can never go stale against a duplicate spec.
  - **Per-rule folder: created on first add/revise, then persistent.** `rule_docs/<id>/` is created when that rule is first added or revised and then **persists**, so a later revision has an obvious, hard-to-miss home (text-only placement guidance is skip-prone — it has misfired in practice). No folder is created pre-emptively for rules never touched.
  - **Three snapshot-distinguishable states** (the persistence boundary differs from `docs/`: `docs/<domain>/` persists via a live Spec, `rule_docs/<id>/` persists via the folder itself):
    - *idle* — exactly one file, `.gitkeep`; valid **only** for a folder mapping to an *existing* rule (`rules/<id>/<id>.md` or `snippets/rules/<id>.md` present). A candidate (no rule yet) is never idle — it is either incubating, being promoted, or (on discard) deleted.
    - *candidate incubation* (a new rule, pre-existence) — `<id>_incubation.md` (the single self-sufficient *planning* home; it may also carry a round-scoped `<id>_work_packet.md` as non-authoritative work evidence — the discriminator from *active lifecycle work* is the presence of `<id>_incubation.md`, not the work packet; per **E3** no `_design`/`_plan`/`_spec` sibling during incubation — those begin only at promotion).
    - *active lifecycle work* (an existing-rule revision, or a candidate's promotion) — one or more of `<id>_design.md` / `<id>_plan.md` / `<id>_work_packet.md` and **no `<id>_incubation.md`** (if `<id>_incubation.md` is present the snapshot stays *candidate incubation* even when a `<id>_work_packet.md` is also present — `<id>_incubation.md` is the state discriminator; at promotion the `_incubation.md` is removed as the candidate-lifecycle closeout — its E4 absorption into **the entry promoted artifact (the `<id>_design.md`)** completed as a precondition of removal (see *Candidate lifecycle*) — within **the promotion transition: the one atomic changeset that removes `<id>_incubation.md` and writes `<id>_design.md` together (the `_incubation.md` → `_design.md` swap)**, so `<id>_incubation.md` never coexists in committed state with `<id>_design.md` / `<id>_plan.md` / `<id>_spec.md` — E3 stays intact (that atomicity guarantees only the non-coexistence; *what state* the promoted artifact is then in is **promoted-but-not-live** — a domain's `prelive` Spec (see *Spec identity*) or a rule's pre-terminal-landing `_design` / `_plan` — not yet implementation authority)). An existing-rule revision is the normal Design → Plan lifecycle, **not** incubation, so E3 does not bind it.
  - **Committed-temporary; deleted at closeout; folder kept.** During the work the applicable committed-temporary docs for the current state (an incubation document and/or a work packet during candidate incubation; `<id>_design.md` / `<id>_plan.md` / `<id>_work_packet.md` during active lifecycle work, whether an existing-rule revision or after promotion) are committed and pushed (git history preserves them); at **closeout** every such document is **deleted** (retire = deletion) — each at *its own* closeout, per the state it belongs to: a candidate-incubation `<id>_incubation.md` (and any `<id>_work_packet.md` carried during that incubation) at the **candidate-lifecycle closeout** (promotion or discard), and the active-lifecycle `<id>_design.md` / `<id>_plan.md` / `<id>_work_packet.md` at the **promoted-lifecycle closeout** — leaving the folder in its *idle* `.gitkeep` state. The persisted folder + `.gitkeep` is a **rule-defined structural anchor**, not an ad-hoc dead / archive / bucket folder (those stay rejected): `retire = deletion` binds the *committed-temporary docs*, while the folder persists by this rule's own definition — but **only an *existing* rule keeps an idle folder.** A **discarded** candidate (closed without promotion) deletes its whole `rule_docs/<id>/` folder (no idle `.gitkeep`); on **promotion** the candidate's folder becomes the now-existing rule's folder (idle once its planning docs are deleted at closeout).
  - **Purity (mechanically checked by `scripts/docs-working-model-check.ps1`).** Under `rule_docs/`: no loose top-level files, no orientation README (the authoritative scope is *this rule*; `docs/README.md` stays the single orientation map). Every child `rule_docs/<id>/` maps 1:1 to a rule or candidate id; its files are only `.gitkeep` or `<id>_{incubation,design,plan,work_packet}.md` (`.md` only); no subfolders, no `archive/` / `consumed/` / `old/` / miscellaneous files. An *idle* (`.gitkeep`-only) child must have a corresponding existing rule (`rules/<id>/<id>.md` or `snippets/rules/<id>.md`) — otherwise it is an **orphan violation** (a discarded candidate's or deleted rule's folder left behind); a child holding `<id>_incubation.md` is a candidate and need not have a rule yet. The check **recognizes the idle state (exactly `.gitkeep`) as valid** and must not flag or remove a `.gitkeep`-idle folder; it verifies *structure* only — the **closeout gate** (a process step, not a snapshot check) enforces the deletion-at-closeout obligation. On rule rename the folder is renamed/migrated; on rule deletion the folder is deleted (no orphan). `docs/` is intentionally looser (it also holds live domain Specs and `docs/README.md`); this purity binds only `rule_docs/`.
- **Candidate lifecycle.** (optional ephemeral **seed** brainstorm — in conversation or out-of-repo scratch, not a required layer) → **incubation anchoring** (the consistency-checked incubation document lands in-repo at its first *approved* commit; the anchor is the meaning of that approved commit, **not** a self-commit licence — commit / push stay separate explicit approvals) → formalize (the incubation document matures to canonical *form*, still **non-authoritative**) → dev/test/pilot. At each **review-date** the candidate is decided: **promote** (a domain candidate enters the *Design / Plan / Spec lifecycle* with canonical filenames; a rule candidate writes its terminal rule file via Design → Plan → rule), **discard** (a closeout — delete), or **continue** (remain in incubation with a **new review-date** — no candidate continues without a live review-date; a review-date that passes undecided leaves the candidate non-conformant, stale until decided). On **promote**, a domain candidate's `docs/<candidate>/` becomes the new domain's home (renamed to the final name if it differs) and a rule candidate's promoted lifecycle produces its `rules/` or `snippets/rules/` rule file as that lifecycle's terminal output; either way the incubation document's current-bearing content is absorbed (E4) — every current-bearing item represented in E4-required form, as applicable (adopted conclusion / rejected alternatives / judgment-changing evidence type / scope / failure criteria / negative evidence), not raw-carried — into **the entry promoted artifact — the `_design.md` that the promotion transition writes** (a domain candidate then continues Design → Plan → Spec, a rule candidate Design → Plan → terminal rule) as a **precondition of removal** within the promotion transition (the atomic `_incubation.md` → `_design.md` swap). This E4 absorption carries the candidate's **decision-grade** content into the `_design` (E4 conclusions are never absorbed into a Work Packet); separately, any **round-scoped** analysis the candidate chooses to keep moves to an active-lifecycle Work Packet per the *Lifecycle invariant*, and **nothing of the `_incubation` is raw-carried**; the subsequent Design → Plan → Spec / rule is the normal promoted-lifecycle progression (its closeout reconciles the eventual Spec / terminal rule 1:1). Then the `_incubation.md` is removed (this removal is the candidate-lifecycle closeout; a promoted artifact smaller than the `_incubation` is not a licence to lose current-bearing content — incomplete absorption is an E4 violation, not a reason to preserve the raw document); each **unresolved incubation open question** is, at promotion, either resolved or explicitly deferred — for a **domain** to the new `<domain>_backlog.md` (one line + a reopen / start condition; or, until that backlog exists, to a `Deferred Questions` section of the entry `_design` that the Plan then absorbs into the backlog), and for a **rule** (which has no backlog) kept in the `_design` / `_plan` and resolved before the terminal rule lands — so none is lost; an open question that is **neither resolved nor deferred** blocks the transition to `live` (closeout / terminal landing), while a deferred backlog item is tracked future work, not a blocker; **for a rule candidate the `rule_docs/<candidate>/` folder is *not* deleted on promotion — it persists as the now-existing rule's planning home (idle `.gitkeep` once any promotion planning docs are deleted at closeout), per the `rule_docs/` purity bullet above.** On **discard** — the discard variant of the candidate-lifecycle closeout, to which the E4 promotion-absorption does not apply — the document is deleted with no absorption into any canonical surface (**and, for a rule candidate, its whole `rule_docs/<candidate>/` folder is deleted — a discarded candidate becomes no rule, so it keeps no idle folder**); the discard rationale (the negative evidence that ended it) is stated in the discard commit message (preserved by git history), so a later re-proposal can find why it was rejected. A **candidate-lifecycle closeout — promotion (renamed or not) or discard —** must, in the same changeset, sweep the name-identity mentions of that candidate carried by canonical surfaces (a previously promoted artifact, a canonical index — its E1 tracking metadata included — or an accepted glossary entry): find them per the *4-class reference sweep*, then correct / remove / re-home each — on **promotion**, update each mention's candidate-status wording (the target is no longer a candidate, so the mention leaves the *Promoted-artifact sibling reference* scope and the normal cross-reference rules govern it; on rename-at-promotion the name is corrected too); on **discard**, correct or remove the mention (this is the **sibling-mention sweep**). The sweep disposes reference **wording** only: whether the candidate's pending terms are finalized, rejected, or carried forward remains the separate per-term decision under the glossary's terminology clauses — the disposition question is raised and checked in the same changeset, never forced to the same outcome as the sweep. (De-promotion — a *State migration* event on a promoted-but-not-live artifact, not a candidate-lifecycle event — carries the same sweep duty via its own clause; see *State migration — De-promotion*.) The incubation document follows the lifecycle form and *Stage rewind* but remains non-authoritative.
- **Distribution-tier landing (universal-core ↔ project-residue split).** When a changeset lands rule content on the global distribution tier (`snippets/rules/<id>.md`) — a rule candidate's terminal landing, or a later revision that lands new content on, or extracts existing content from, a distributed rule — that changeset also completes the **universal-core ↔ project-residue split**: the distributed rule file carries only content that passes the distribution tier's own admission criteria (`snippets/rules/README.md` — the single home of those criteria, not restated here), and every project-residue item the changeset puts in play — **content the lifecycle carried toward the rule file, or existing residue the revision extracts from it** — is, in that same changeset, either re-homed to its correct repo-side surface (e.g. the repo-only `rules/` tier) or explicitly discarded with the rationale recorded in the commit message — never silently dropped when the closeout later deletes the planning docs.
- **Glossary registration is independent of incubation.** Terminology changes in `rules/terminology-glossary.md` have their own cycle and several sources (an existing-feature/rule change, an operating-concept change, recording a do-not-revive (rejected) entry, an accepted-term owner-boundary adjustment, or naming cleanup); a candidate's incubation is only one such source. The *Proportionality rule* governs a glossary edit as it governs any doc edit, within the glossary's single-home constraint. **Meaning-home division:** the glossary is the single home of a term's *one-line meaning + classification*; the term's *full semantics / procedure* live on its **finalization-owner** — the owner-bearing surface responsible for that term's semantics and finalization (a live Spec, a rule, or a deployed implementation; a live Spec may be a finalization-owner without being reclassified as an active-behavior `owner surface`). A *finalization-owner close* is that surface (not the glossary) reaching its target state at closeout; when a term is finalized at such a close (see below), the glossary records its one-line meaning + classification at that point. A term not yet finalized stays `pending` while its finalization-owner is **not yet at live authority**, and `owner-pending` once its finalization-owner is **already at live authority** (a landed rule, a deployed implementation, or a live domain whose Spec is `sync-required`) with finalization deferred to that owner's close — an `owner-pending` term carries a live one-line meaning + classification even though it is not yet final.
- **Terminology registration is conditional (not mandatory-at-anchoring) — and this conditional thin-reservation clause governs terms *a candidate introduces*.** A `pending` term from any other source (the *Meaning-home division* above names several — a candidate's incubation is only one such source) is outside this clause's scope; what form such an entry takes within the *Meaning-home division* cap is addressed, as an explicitly open question, in the *Owner-pending registration* bullet's form note below. A *meaning-bearing* term a candidate introduces is registered in the glossary as a thin `pending` reservation **only when it is exposed or collision-prone**:
  - *exposed* — the term appears on a tracked surface other than the candidate's own incubation document (e.g. another doc's filename / heading / anchor, a reference from another document, or a rule / spec / operating doc); the candidate's own filename, headings, anchors, and internal references are not exposure.
  - *collision-prone* — an **observed** identical / confusable use of the name, either against a glossary-registered term (accepted — with or without an owner boundary — / pending / owner-pending / rejected) or against another candidate's term introduced in the same changeset (even if neither is registered yet). The *collision class* is the matched target's category (which glossary state, or a same-changeset candidate).
  Terms used only inside the incubation document (internal labels, one-off phase names, explanatory nicknames) are **not** registered — unless the name is collision-prone per the definition above (a collision against a glossary-registered term or a same-changeset candidate is itself a registration trigger, even for an otherwise-internal term). The candidate author may proactively register a thin reservation whenever exposure is observed (when an exposure's collision class is still uncertain, treat it as an exposure-only reservation: reserve thin, no `collision-note`, define no meaning). A reviewer may require a reservation on either basis — for exposure, citing the observed external surface; for collision, citing the observed surface plus the collision class — a bare "could be confused" is insufficient. Classification disputes are arbitrated by the glossary rule owner.
  - **Reservation fields (use only these, no others):** `candidate` (the candidate that introduced the term) / `facet` (which sub-aspect, a few words) / `not-this` (a few-word exclusion marker) / `eventual-owner-surface` (the current expected owner surface where the term will ultimately live — optional if not yet known; naming the destination is not a finalize-at-promotion); plus `collision-note` **only** for a collision-prone reservation (fixed form "Potential collision with X; defines no semantics; see this reservation's candidate"), omitted for an exposure-only reservation. **"Define no meaning"** = no positive one-line definition, example, usage, procedure, or schema; `facet` / `not-this` are short identity-scoping markers (a few words each), and if either grows into a positive definition / example / procedure it violates "define no meaning". No durable path pointer to the `_incubation` document — owner identity only (an owner / surface name, never a file-path pointer; per *Durable-pointer prohibition* / E2). The full domain-local definition stays in the incubation document.
  - **Name reservation vs meaning finalization.** Registering a thin `pending` reservation (a *name-level* act) is distinct from *finalizing a term's meaning*. The glossary finalization outcomes are exactly **accepted** / **accepted-with-owner-boundary** / **rejected**. During incubation only the conditional name reservation may be required; **final terminology is never required during incubation.** Each finalization-owner close forces a per-term decision for every meaning-bearing term that surface **owns** (i.e., is the finalization-owner for — not terms it merely references), whether already registered `pending` or first going public at this close: either finalize it (to one of the three outcomes) **or** explicitly carry it forward as `owner-pending` with a stated reason and a next close-condition (finalization is **decoupled** — a term may legitimately need more incubation than one finalization-owner close, and a candidate may promote while one of its terms stays `pending` / `owner-pending` or is rejected). The close is a decision point, not a mandatory-finalize point. When a term *is* finalized in a changeset, its glossary finalization is applied **before that change's corrected-state review** (so the review validates the final terms; applying it after would make the review stale). When a term is finalized to **`rejected`** (from `pending` or `owner-pending`), the change must first sweep that term's accepted-looking usage across **all** surfaces (the *4-class reference sweep*) and remove / reword it or mark it non-canonical ordinary wording **before** the corrected-state review — a rejected term must not survive as an accepted-looking heading / token anywhere outside the glossary's *Rejected terms* section. The finalization-owner close is: a domain candidate → its Spec / implementation closeout changeset; a rule candidate → its terminal rule-file landing changeset; an existing-rule or live-domain revision → that revision's closeout (which may be deferred); a non-candidate terminology cleanup → that change's own corrected-state review.
  - **Owner-pending registration (finalization-owner go-live).** A `pending` reservation covers a term whose finalization-owner is **not yet at live authority** (a pre-promotion candidate is the typical case). When a term's finalization-owner instead reaches or updates **live authority** with finalization still deferred — an existing-rule revision's rule landing (the rule is authoritative on landing while its closeout is a separate, possibly deferred event; a *rule-candidate* promotion's terminal landing is instead itself the finalization-owner close, so its terms meet that close's per-term decision there — finalized, or explicitly carried forward as `owner-pending` — rather than being registered `owner-pending` by the landing itself), a deployed implementation, or an existing live domain's in-place Spec update (which enters `sync-required`; a *new* domain candidate's Spec is a pre-closeout blueprint and does **not** trigger this) — the term is registered, or moved from `pending`, to **`owner-pending`**. The trigger is the live-authority landing of the finalization-owner, **not** the commit / push approval act. This does **not** change the content rule for `pending` entries themselves, and **the `pending` ↔ `owner-pending` split is only the *finalization-owner-live* axis** — a status split that decides no entry's form. A `pending` entry's form is governed **in part**: the *Meaning-home division* above caps every glossary entry at a one-line meaning + classification, and the conditional-registration clause above fixes a **candidate-introduced** `pending` term to the thin reservation form ("define no meaning"). **Whether a `pending` term from a non-candidate source stays thin or may carry a one-line meaning within that cap is a question this rule deliberately leaves open** — a named open question (capped by the *Meaning-home division* either way), to be settled as a separate normative decision at the first actual non-candidate `pending` registration (not settled by this rule's landing, and not to be inferred from the candidate clause). An `owner-pending` entry carries only: a one-line meaning, `finalization-owner` (an owner id or a **tracked owner-surface path** — never a durable pointer to an `_incubation` document, `log/**`, or out-of-repo scratch, per *Durable-pointer prohibition* / E2), `facet`, `close-condition` (its finalization-owner's close), and `not-this` (plus, **only** when a term is *carried forward past* a finalization-owner close without being finalized, a short `carry-forward reason`); no procedure, full semantics, or runtime state. An owner-pending entry registered at go-live (before its first finalization-owner close) needs no carry-forward reason. Finalization is **monotonic**: a normal lifecycle transition never moves `owner-pending` back to `pending`; the sole exception is correcting a **wrong go-live judgment** (the finalization-owner was not actually live), which returns the entry to `pending` status, dropping `finalization-owner` and `close-condition` (for a **candidate-introduced** term this restores the thin reservation form, dropping the one-line meaning; for a non-candidate term the correction decides *status* only — its form stays within the open form question stated in the form note above). If a finalization-owner is **retired / removed / replaced before its close**, that retirement is itself a finalization-owner change that forces the per-term decision (finalize — `rejected` if the meaning is withdrawn, or re-home `finalization-owner` if the meaning moved to another live surface), so no `owner-pending` term is left orphaned.
  - **Transition (already-anchored candidates).** This model takes effect at this rule's landing. A candidate already in-flight then (the existing `docs/consultation/`, `docs/blind-advisory/`, and `rule_docs/subagent-work-orchestration/` candidates) is **not** retroactively in default. A *realigning changeset* is the next changeset that edits that candidate's content; such a changeset must conform the candidate to this model (a changeset may not edit the candidate's content while leaving it non-conforming). An unrelated change (one not touching the candidate's content) does not end the exemption. The candidate's terms become fully bound at that realigning changeset's closeout, where the per-candidate exemption ends (it is not permanent). The mechanical check **for this conditional terminology-registration model** — the future terminology-registration check to be added to `scripts/docs-working-model-check.ps1` — is enabled in that same realigning changeset (transition-aware), not at this rule's landing; the script's **current structural E1/E2/E3 (and EN-2) checks** are 'rule requirements now' (per *Form early, authority late*) and bind immediately, not under this transition deferral.
- **Form early, authority late (the core invariant).** A candidate artifact may use canonical *form* but holds no canonical *authority* — a candidate's formalized content is never interpreted as a §*Spec identity* canonical Spec. Header text alone does not hold this; E1–E5 below bind as **rule requirements now** (conformance is manual until their checks exist — the checks are a separate implementation change):
  - **E1** — domain/rule discovery is by promoted canonical artifact, never by `docs/<candidate>/` or `rule_docs/<candidate>/` existence (a folder holding only `_incubation.md` is a non-domain/non-rule candidate container; it is not an *End-state placement* home and not a discovery target before promotion). No new central registry — at most thin candidate-tracking metadata (name / owner / review-date) on an existing index/manifest, insufficient by itself to locate or use the candidate as a canonical input or discovery index (candidate tracking, not the durable reference E2 forbids). Discovery is **two-layer**: a promoted canonical artifact — *including a `prelive` Spec* — is **governance-discoverable** (visible for review / lifecycle tracking), but **implementation authority** (the 1:1 basis for live behavior, and the default a consumer builds on) belongs only to a `live` / `sync-required` Spec, or for a rule the **landed terminal rule file**; a `prelive` (or otherwise promoted-but-not-live) artifact's mere existence is *not* authority, so its presence must not be consumed as live behavior.
  - **E2** — canonical rules/indexes must not durably reference a candidate `_incubation` document (its formalized content included); a canonical→candidate reference to the candidate's document or its content is admissible only as an **absorbed-conclusion summary** (satisfying E4, re-reviewable without the candidate path/link) — this clause governs such document-level references; a **name-identity mention** of a still-incubating sibling candidate is governed by the *Promoted-artifact sibling reference* clause below, neither licensed nor forbidden by this one.
  - **E3** — before promotion a candidate artifact is not a default or input of any canonical surface (rules, indexes, templates, skills, reviewer checklists, Work Packet generator/input); no canonical-looking sibling (`_design` / `_plan` / `_spec`) is created **while the `_incubation.md` exists** (the incubation period). The promotion transition is the boundary that atomically removes `_incubation.md` and writes `_design.md`, so a sibling appears only *after* incubation ends, never *during* it (the binding window is the incubation period, not the whole promoted lifecycle). The prohibition is by **lineage**, not just same-folder name: while the `_incubation.md` exists, no `_design` / `_plan` / `_spec` may be created for that candidate, its successor id, or its rename target — and the promotion transition carries the source `_incubation` disposal (or its rename to the final name) and the target `_design` creation in the **same changeset**, closing the rename-evasion path (an old `_incubation` left beside a new-name `_design` is the same violation).
  - **E4** — absorption into a tracked file carries: adopted conclusion / rejected alternatives / the evidence type that changed the judgment / scope / failure (discard) criteria / known negative evidence — so "why this survived" is re-reviewable without raw links.
  - **E5** — this rule's own incubation-tier addition, **and its generalization from pre-domain to pre-promotion (adding rule candidates)**, is a one-time bootstrap (incubation cannot incubate itself), not a precedent for later candidates.
- **Promoted-artifact sibling reference.** A promoted canonical artifact — at any promoted phase (its entry `_design` and subsequent `_plan`, a `prelive` or live / `sync-required` Spec, or a landed terminal rule), and equally a canonical index or an accepted glossary entry — may mention a **still-incubating sibling candidate** (*sibling* here = a fellow pre-promotion candidate of either kind, not the canonical-role file sense used elsewhere in this tier) by **name identity only**, and only under all of: (a) never a path / durable document reference (E2 unchanged) and never consuming the candidate's semantics as a default or input (E3 unchanged); (b) the referencing sentence itself keeps the target's status honest — it names the target *as a candidate* (non-authoritative, pre-promotion), so the mention cannot be read as discovery of a live surface (E1's two-layer discovery is unchanged: a name mention confers neither governance discovery nor implementation authority), and such a status-honest mention does not breach the candidate header's `not referenced by canonical rules/indexes` marker, which keeps its E2 / E3 sense (no durable document reference, no input-consumption); (c) the form is prose and per-reference — no marker vocabulary, registry, or index is introduced for these references, and a set of such mentions must not function as a candidate discovery index; E1's thin candidate-tracking metadata on an existing index / manifest stays a separate, unchanged allowance (this clause governs prose mentions, not that tracking row); (d) what the promoting candidate's own E4 absorption carries into its promoted artifact stays bounded to the current-bearing content that artifact's own identity / contrast needs — a sibling mention is identity-scoping, never a restatement of the sibling's semantics (the *Cross-domain semantics restriction*'s interface-not-semantics principle applies unchanged). A mention admitted by this clause is a point-in-time status claim; the sweep duties at the target candidate's life events (*Candidate lifecycle*; *State migration — De-promotion*) keep it true.
- **Incubation applicability (default strict).** An incubating candidate is governed **by default by every rule in this document**; only the items listed here as relaxed are relaxed (an unlisted rule stays strict, and adding a relaxation is itself a reviewed rule change). This is an applicability statement, not a loose exception zone. **Strongly applied:** public-safe boundary, the *Durable-pointer prohibition*, placement / filename (*Stable filename rule*), the non-authoritative marking, owner / review-date / discard criteria, E1–E5, and confusion-prevention with existing domains / rules. **Not required during incubation:** closeout 1:1 sync, accepted-term do-not-repeat strictness, production polish, final terminology. **Still required (so the candidate stays judgeable at its review-date):** the identity, scope, not-this, contrast, and discard evidence the promote / discard decision needs — strengthening judgeability must narrow the candidate's *own* identity / scope / not-this, and may not require expanding any foreign domain's definition / schema / procedure.
- **Data separation (seed / log / tracked).** A candidate's **seed** is an *optional, ephemeral* brainstorm (conversation or out-of-repo scratch), advisory only and never a durable pointer (*Durable-pointer prohibition*) — it is **not a required layer**, and once the work reaches incubation anchoring the in-repo incubation document is the self-sufficient home, so no separate seed document need persist; measurement accumulates under `log/**` (gitignored, never git-tracked); a tracked file carries only self-contained absorbed decisions. Candidate planning while incubating is committed-temporary, never a permanent shadow of any active surface.
- **Incubation vs Work Packet.** Different axes, not mutually exclusive. A Work Packet is a round-scoped *work* artifact (investigation, line-level analysis, evidence notes) for whatever round-work is underway; incubation is the candidate *lifecycle stage* where the promotion target (a domain or a rule) does not yet exist (deciding whether it should). A candidate may therefore carry a round-scoped Work Packet while still incubating; what it must not create before promotion is a canonical `_design`/`_plan`/`_spec` sibling (E3).
- **Absorption is a commit-boundary crossing.** Moving seed/scratch content into a tracked file is subject to the same content scrutiny as any other committed change — the absorption step is not a side channel that bypasses the repo's commit gate.
- **No round cap.** Incubation has no fixed cycle/round limit; it ends on convergence or on the operator/user judging the candidate ready or dead — never on a count.

## Future-work queue (`<domain>_backlog.md`)

- One file per domain. Each item = **one line + a reopen/start condition**. No narrative, no incident logs, no closeout reports.
- **Closed items: delete the row** (preservation = git history). Exception — a one-line tombstone is kept **only when** a live inbound reference to the closed ID remains and cannot be rewritten as a direct decision statement; the exception is determined by the bare-token/ID class of the reference sweep, not by discretion.
- **ID reuse prevention**: the file header carries one line `next ID: <PREFIX>-NN`, monotonically increasing (never decreased or reused after row deletion). `<PREFIX>` continues the domain's established ID prefix convention (BR / RV / IU / SK …); a new domain picks a short domain abbreviation.

## Spec ↔ implementation 1:1 synchronization

- The unit of 1:1 is the **normative sentence**, not lines or code (a Spec is never a prose mirror of scripts).
- Direction 1: every normative sentence in the Spec must be verifiable in the implementation.
- Direction 2: every externally observable behavior and ownership boundary of the implementation must have a corresponding Spec sentence. Internal implementation detail (function decomposition, naming, algorithm choice) is not a 1:1 target.
- **Reconstructibility** = from the Spec alone, an implementation with the *same behavior* can be rewritten (not the same code); from the implementation alone, the Spec's *normative meaning* can be reconstructed (meaning-level, not sentence-mirroring).
- Operating test: if an implementation change requires no Spec-sentence change, it is a refactoring (no sync needed); if it does, it is a behavior change (Spec update mandatory).

## Live-Spec update — the sync-required transition

For further development on a domain with a live Spec: after Design → Plan, **update the live Spec in place to the new target state** (no copies, no delta documents — single home). From that moment the domain state is **sync-required** (marked in the Spec's lifecycle-state section); implementation catches up; closeout re-verifies 1:1 and returns the state to **live**. `sync-required` is for a domain whose Spec was **already live**; a newly-promoted domain's first Spec is **`prelive`** (promoted but never yet live — see *Spec identity*), a different marker, not `sync-required`.

## Proportionality rule

Edits that do not change normative meaning (typos, stale pointers, wording cleanup) may be applied directly without Design/Plan — the normal review gate still applies. The moment normative meaning changes — an allow/forbid boundary, a behavior statement, an owner mapping, a validation expectation — the full lifecycle is required. The arbiter is the operating test above.

(Axis note: the proportionality test decides *whether the Design → Plan → Spec lifecycle is invoked* for a change — a normative-meaning change invokes it, a meaning-preserving edit does not (the Work Packet, incubation, and backlog have their own creation triggers, not governed here). The *Lifecycle invariant* — detail flows downward — decides *what each created lifecycle artifact holds*. Different axes: a change can be normative (lifecycle required) and still must keep each artifact at its own altitude; and omitting a lifecycle artifact does not license moving its content into another artifact.)

**Abuse guard** — so "wording cleanup" cannot smuggle meaning changes: (a) a direct edit's change description must state it is **meaning-preserving**; (b) if editor and reviewer disagree, or doubt remains, escalate to the lifecycle.

## Closeout — reduced two-level gate

A change's docs closeout is not done until both levels pass. **Inspection and reporting are unconditional; only updating is conditional.** For every listed surface the closeout report states `updated: <file> — <what>` or `checked: <file> — no change required`; silently skipping a listed surface is a gate failure.

- **Level 1 — orientation**: `docs/README.md` (routing/order reflects the change and the current mixed state), plus any not-yet-retired legacy orientation surface the change affects.
- **Level 2 — domain-local**: the domain's `<domain>_spec.md` (1:1 sync verified; lifecycle state updated) and `<domain>_backlog.md` (rows added/closed per the queue rule). For a not-yet-migrated domain, its current authoritative surfaces stand in until its migration batch. For a `rule_docs` / terminal-rule change, the Level-2 surface is the terminal rule file itself (`rules/<id>/<id>.md` or `snippets/rules/<id>.md`), reconciled 1:1; a rule has no `<domain>_spec.md` / `<domain>_backlog.md`, so its open questions are resolved before the terminal rule lands (no backlog deferral).

## Lifecycle closeout — absorption and retire

A Design/Plan/Spec lifecycle closeout is not done until **all** hold:

- Spec ↔ implementation reconciled 1:1 (normative-sentence correspondence recorded as evidence in the closeout report / `log/evidence/**`, not in checklists).
- Every current-bearing Design decision and still-relevant Plan decision is expressed in the Spec (or the correct owner surface); no unique live meaning remains only in the Design / Plan / Work Packet.
- Inbound references are updated / removed.
- The Design and Plan are **retired** — retire is **deletion** (repo lifecycle); git history is the preservation mechanism, never an archive / `consumed/` folder.
- The Work Packet is deleted.
- This is the **promoted-lifecycle closeout** (it disposes the Design / Plan / Work Packet above, subject to the applicable closeout gates). A candidate's `_incubation.md` is **not** listed here because it was already disposed at the earlier **candidate-lifecycle closeout** (its promotion or discard — see *Candidate lifecycle*). This **locally disambiguates** *closeout* and does **not** redefine the word elsewhere (the *reduced two-level gate* docs-closeout, the *finalization-owner close*, and discard-as-a-closeout keep their meanings): the candidate-lifecycle closeout precedes the promoted-lifecycle closeout, and the *candidate-discard closeout* is simply the discard variant of the candidate-lifecycle closeout. Each temporary artifact is disposed at **its own** closeout.

## Stage rewind

- **Plan violates the Design** → stop, redesign the Design, restart the Plan.
- **Spec violates the Plan** → stop, re-plan, restart the Spec.
- **Implementation exceeds the Spec boundary** → stop and ask the user; never silently widen scope.

## State migration (same role-slot)

- Beginning a new revision in a role-slot whose *prior* revision's planning docs (`_design` / `_plan` / `_work_packet`, or a candidate `_incubation`) remain un-retired requires their **disposition first** — either the applicable closeout, or an explicit decision to continue the *same not-yet-closed-out work* in that role-slot under this revision (not a revival of already-retired docs, and never a bypass of the closeout-deletion obligation).
- A carried-over / pre-revision artifact is **non-authoritative until judged** — reuse / re-verify / discard. This binds only the **same role-slot**; it does not block parallel per-domain / per-rule revisions, and it creates no `archive/` / subfolder / extra file (per *Stable filename rule* and the `rule_docs/` purity).
- **De-promotion (a promoted-but-not-live artifact only).** A **promoted-but-not-live** artifact — a domain's pre-closeout promoted lifecycle (`_design` / `_plan`, and/or a `prelive` Spec once written), or a rule's pre-terminal-landing `_design` / `_plan` — may be withdrawn by an explicit **`promotion-withdrawal` changeset** that disposes **all of its promoted-lifecycle artifacts** (`_design` / `_plan`, and a `prelive` `<domain>_spec.md` if one was already written — so E1 stops discovering a promoted Spec for a withdrawn candidate) and re-opens the candidate's `_incubation` (carrying a `restored from withdrawn promotion` marker; the candidate id may be reused and its history is preserved) — a **recorded** reversal, never a silent un-recorded rollback. This is **history-preservation, not artifact-identity monotonicity**: candidate revival *is* allowed, only un-recorded rollback is forbidden. Once the artifact is **live** (a domain's closeout-reconciled Spec, or a landed terminal rule), de-promotion is closed — changing it then goes through the normal lifecycle (repeal / supersede), not withdrawal. A withdrawal changeset also performs the **sibling-mention sweep** defined under *Candidate lifecycle*: name-identity mentions of the withdrawn candidate carried by other canonical surfaces are corrected / removed / re-homed in the same changeset — and inbound **document-level references** to the withdrawn promoted artifacts (legitimate while they were promoted — e.g. a reference to the withdrawn `prelive` Spec) are corrected / removed in that same changeset as well, so none survives the reversal as an E2 violation or a dangling reference.

## Stable filename rule

- Domain documents use **domain-prefixed role filenames**: `<domain>_design.md` / `<domain>_plan.md` / `<domain>_spec.md` / `<domain>_backlog.md` / `<domain>_work_packet.md` (the last is the class-2 temporary work document — a lifecycle role filename existing only during a change, deleted at closeout; not an auxiliary role doc). A pre-promotion candidate additionally uses `<candidate>_incubation.md` (*Incubation tier* — likewise a class-2 committed-temporary lifecycle role, deleted at its **candidate-lifecycle closeout** — promotion or discard), located at `docs/<candidate>/` for a domain candidate or `rule_docs/<candidate>/` for a rule candidate. Re-creating after deletion reuses the same role filename. Forbidden: `<topic>_*.md` topic-named files, filename-evading subfolder splitting (e.g. `docs/<domain>/work/`), per-feature design/plan/spec proliferation inside one domain.
- Auxiliary role docs (`_policy` / `_contract` / `_state` / `_status` / `_guide`) are **deferred** — not created by default; introduced only by an explicit Design/Plan decision.
- **Package-local form vs domain form.** This package's `templates/` / `checklists/` files carry the package prefix `docs-working-model_` and a `_template` / `_checklist` role suffix: they are **forms that produce another domain's** documents, not a domain's own artifacts.

## Authoring language

- New human-facing docs are **Korean by default**; technical identifiers (file / function / flag names, code, CLI tokens) stay English. This is a repo-development authoring convention for this repo's own docs — an adopter project chooses its own language.

## Domain-local closure and top-down reference

- **Domain-local closure** — each domain document is understandable from its own folder + its own live Spec + its own active surface + the stable interfaces it explicitly depends on. If understanding it requires reading another domain's semantics, the Spec has failed.
- **Top-down reference** — root `README.md` → `rules/README.md` → this rule package → a domain folder → the domain Spec. A lower layer does not complete its meaning by holding a durable pointer up to a routing document.

## Cross-domain semantics restriction

- Domain↔domain **semantics** references are forbidden by default. Narrowly allowed: a stable path / interface, schema, contract boundary, marker / payload-root / lifecycle boundary, or a domain whose essence is a cross-domain mechanism (install-update) — and even then only the **interface**, never a restatement of the other domain's semantics.
- Test: "if the target domain's implementation changes, does this reference change too?" yes = semantics (forbidden); no = interface (allowed).
- **Incubating candidates — define-by-contrast, foreign-semantics pointer-only.** An incubating candidate may state its identity by contrast with an existing concept but must not own the other's semantics. *Allowed:* "unlike X, this candidate covers only Y"; "X's authoritative meaning follows X"; "this candidate issues no verdict". *Blocked (these stay reviewer blockers):* restating another domain's full status vocabulary, its lifecycle / permission / completion semantics, or its normative behavior; a foreign schema / field / procedure written as a standalone definition with no source pointer. The required non-authoritative marker (a candidate declaring its *own* authority status) is allowed; what is banned is a *global self-exempting* meta-claim about the doc's own cross-domain behavior (e.g. "this document does not restate any other domain") — unverifiable boilerplate, often self-contradicted by the doc's own contrast statements. A genuine cross-domain restatement defect stays a blocker; what is forbidden is *requiring more foreign definition* to fix it — fix it by narrowing the candidate's own claim or adding an owner / surface identity pointer (a name, never a file-path pointer). Inter-candidate relations are marked thinly (`depends-on` / `contrasts-with` / `independent-of`) on the candidate surface, not as glossary reservation fields, with foreign semantics pointer-only (preserves discard independence).

## On-demand status-briefing model

There is **no committed project-current mirror file** — no committed active queue and no committed project-current summary. "What is done / what remains / what should I do next" is answered **on demand** by the agent reading the authoritative surfaces — after migration, each domain's `<domain>_spec.md` (lifecycle state) + `<domain>_backlog.md`; for a not-yet-migrated domain, its current authoritative surfaces stand in until its migration batch — then synthesizing a conversational briefing. The user selects the next task conversationally; no mirror file is written.

## Template / checklist conformance gate

- Each produced Design / Plan / Spec / **Work Packet** must pass its corresponding checklist, a lifecycle closeout must pass the closeout checklist, and a **promotion-boundary / promoted-but-not-live event** (promotion / discard / de-promotion) must pass the **promotion checklist**. Checklists test **meaning, not section presence** ("is the target state stated in normative sentences", "is round-scoped content zero", "are boundaries standing rather than per-round"). A judgment is recorded as "충족/미충족 + one-line evidence" — and **evidence is not accumulated in checklist bodies**: execution/closeout evidence belongs to operator reports / closeout reports.
- Templates and checklists must not generate boilerplate disclaimers or work-control prose; an approval boundary is stated **once per document**, never per section.
- The package forms live at `templates/docs-working-model_design_template.md` / `_plan_template.md` / `_spec_template.md` and `checklists/docs-working-model_design_checklist.md` / `_plan_checklist.md` / `_spec_checklist.md` / `_work_packet_checklist.md` / `_closeout_checklist.md` / `_promotion_checklist.md`.
- A produced Spec is **not itself operative authority** over the active surface it specifies — the active surface owns behavior (root *Final hard rule*); the Spec specifies it and is reconciled against it.

## Scope of an application + review gate

- This rule defines the model; applying it to any specific legacy surface (domain migration, structure retirement) is its own scoped batch with owner absorption proof + reference sweep + review gate. The rule itself approves no commit / push / publish / merge / release, and no global/user filesystem mutation.
- Any change governed by this model goes through the normal Codex review gate (the globally installed review pipeline, corrected working tree). A verdict (`yes` / `no` / `yes with risk`) approves none of the above.

This is a repo-development rule for this repo only; it is **not** adopter-universal and **not** part of the global distribution (an adopter project may use a different docs authority model). It is not duplicated into `snippets/rules/`. Historical rationale and the predecessor model are preserved in git history.
